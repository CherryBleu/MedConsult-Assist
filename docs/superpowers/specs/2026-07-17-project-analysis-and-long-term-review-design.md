# MedConsult-Assist 项目分析与长期 Review 治理设计

> 日期：2026-07-17  
> 分支：`fix/feat-merge-and-backend`  
> 适用范围：`docs/`、`frontend/`、`backend/`、测试与工程化配置  
> 工作边界：本轮只进行分析、关键路径验证与长期规划，不实施缺陷修复

## 1. 背景与目标

本设计用于把一次性的全栈审查转化为可持续、可复核的治理流程。审查必须同时回答以下问题：

1. `docs/` 是否能作为可信的业务、接口、数据与架构基线；
2. 前端是否覆盖文档要求的角色功能，UI 是否一致，交互是否可用、可访问、可恢复；
3. 后端是否满足接口契约、安全、事务与可靠性要求，是否存在可降阶的时间或空间复杂度；
4. 文档、前端调用和后端实现是否三方一致；
5. 测试、Playwright、运行指标与 review 能否形成持续防退化门禁；
6. 已有计划和已经完成的工作能否被复用，避免重复实施。

本设计不把“发现很多问题”视为完成。完成标准是：每条正式缺陷都有证据、风险、责任域、依赖、验证方式和关闭条件；每次文档修订都有充分凭证；每个阶段都能通过自动化或运行数据判定是否可以退出。

## 2. 已确认的决策

| 决策项 | 选择 | 理由 |
| --- | --- | --- |
| 本轮终点 | 只分析并产出计划 | 用户明确选择 A；不修改业务代码或既有项目文档 |
| 计划周期 | 8–12 周 | 足以覆盖基线、性能、UI/UX、测试与治理，不把架构遗留压缩成不可执行清单 |
| 执行资源 | AI 主导单线 | 以依赖顺序串行实施，必要时只读并行扫描，降低共享文件冲突 |
| 总体路线 | 风险与证据优先 | 先修正尺子和测试地基，再处理安全、性能、体验和工程治理 |
| 文档裁定 | 需求为业务基线，代码/DDL/运行结果可证明文档过时或缺漏 | 防止“为了迁就实现而改需求”，同时允许有证据地修正文档事实 |
| 前端视觉方向 | 本计划不擅自裁定 | 响应式、无障碍、反馈和一致性属于客观门禁；最终视觉风格须在实施时由用户确认 |
| Git | 规划文档独立提交，不 push | 遵守 `AGENTS.md` 的“勤 commit，少 push” |

## 3. 范围与非目标

### 3.1 本设计覆盖

- `docs/修改建议.md` 第一节问题总览的逐条状态核对；
- `需求文档.md`、`接口文档.md`、`数据库设计文档.md`、`架构设计文档.md` 的交叉一致性；
- `frontend/` 的角色功能、路由权限、API 调用、UI 设计系统、响应式、无障碍、状态反馈和 Playwright；
- `backend/` 的 Controller → Service → Mapper、Feign、Redis、MQ、事务、幂等、索引、缓存、分页和复杂度；
- 文档接口 ↔ 前端 API ↔ 后端 Controller 的契约矩阵；
- 单测、流程测试、E2E、构建、CI、可观测性和局部压测方案；
- 已有规格与计划的复用、冲突和状态同步。

### 3.2 本轮不做

- 不修改业务代码；
- 不直接修正现有需求、接口、数据库或架构文档；
- 不执行破坏性数据操作或大规模压力测试；
- 不把未提交的 `backend/medical-record-service/src/test/java/com/medconsult/medicalrecord/MedicalRecordFlowTest.java` 当作既有测试覆盖；
- 不新增 docs 之外的业务接口；
- 不在未经视觉确认时确定新的前端主题、色彩、布局语言或品牌风格。

## 4. 事实模型与证据规则

### 4.1 事实来源的职责

不同事实源回答不同问题，不能简单用单一优先级覆盖所有冲突：

| 事实源 | 负责回答 | 不能单独证明 |
| --- | --- | --- |
| `需求文档.md` | 业务目标、角色、功能边界、合规要求 | 具体接口已实现、运行性能达标 |
| `接口文档.md` | 对外路径、方法、参数、响应、错误码 | 实际 Controller 可调用、代码行为正确 |
| `数据库设计文档.md` | 逻辑数据模型、字段语义、约束与索引意图 | 当前部署 DDL 已同步 |
| `架构设计文档.md` | 服务边界、内部调用、缓存、MQ、降级原则 | 实现已经落地或运行有效 |
| 生产代码与 DDL | 当前实现事实 | 实现符合原始业务意图 |
| 测试与运行结果 | 特定环境下可复现的行为 | 未覆盖场景也正确 |
| `修改建议.md` | 已知问题总览和 review 第 0 步 checklist | 当前状态仍然有效 |
| 复盘、报告、已有计划 | 历史决策和实施意图 | 计划内容已经完成 |

### 4.2 证据等级

| 等级 | 标准 | 使用规则 |
| --- | --- | --- |
| E3 运行证据 | Playwright、测试输出、接口实调、SQL `EXPLAIN ANALYZE`、指标或压测结果 | 可确认缺陷、确认改善或关闭风险 |
| E2 交叉静态证据 | 至少两类独立证据，例如前端调用 + Controller、Service 查询 + DDL、代码 + 文档另一章节 | 可进入正式缺陷台账，但性能影响仍需基线量化 |
| E1 单点静态证据 | 一个代码位置、注释、单份文档或单个搜索结果 | 只能作为调查候选，不能直接改文档状态 |
| E0 推测 | 没有可复核凭证 | 不进入缺陷台账，只能生成探索任务 |

正式缺陷最低要求为 E2。若结论描述具体运行影响、P95、吞吐、内存峰值或锁超时，则必须升级到 E3。

### 4.3 文档偏差分类

| 类型 | 判定 | 处理方式 |
| --- | --- | --- |
| 文档缺漏 | 合理实现和调用均存在，但文档未记录 | 提出补文档建议，附实现、调用或测试证据 |
| 文档过时 | 文档状态描述落后于已提交实现 | 更新状态与截止日期，附代码/DDL和测试证据 |
| 文档内部矛盾 | 同一或不同文档对同一事实有不同定义 | 不擅自选择，按业务意图、实现成本和兼容性给出推荐裁定 |
| 代码偏离文档 | docs 明确要求，但实现不满足 | 原则上创建代码修复任务，不通过改 docs 掩盖缺陷 |
| 产品待决策 | docs、代码或文案无法确定唯一业务规则 | 设置决策门禁，不实现臆想接口 |

每次文档状态变更必须同时满足：

1. 至少 E2；
2. 能定位到 `file:line`；
3. 区分已实现、部分实现、未实现、不适用和证据不足；
4. 未提交 WIP 不得作为“已实现”证据；
5. 若修改优先级或业务方案，必须同步需求、接口、数据和架构中的相关章节。

## 6. 长期治理架构

### 6.1 阶段顺序

采用“风险与证据优先”的六阶段结构：

```text
阶段 0：事实校准与缺陷台账
  ↓
阶段 1：测试、测量与可观测性地基
  ↓
阶段 2：安全、审计与核心业务闭环
  ↓
阶段 3：后端性能与复杂度治理
  ↓
阶段 4：前端响应式、可访问性、反馈与视觉一致性
  ↓
阶段 5：契约、CI、回归与文档收敛
```

阶段 0–1 是其余阶段的前置条件。阶段 4 的响应式、无障碍和错误恢复可以直接规划；视觉风格改造必须先提供原型或对比并由用户确认。

### 6.2 每个实施批次的固定闭环

每个批次必须执行同一流程：

1. **第 0 步**：逐条核对 `修改建议.md` 总览，不跳项；
2. **建立基线**：记录测试、SQL、远程调用、P95/P99、截图或可访问性结果；
3. **最小实现**：一个批次只解决一个内聚主题；
4. **即时 review**：安全、性能、逻辑、潜在问题四维审查；
5. **实测**：前端用 Playwright，后端用测试/接口/SQL/局部压测；
6. **文档闭环**：更新总览状态、契约和复盘；
7. **独立 commit**：验证通过后立即提交，不 push。

### 6.3 Review 深度

- 每个实施批次至少进行 2 轮 review；
- 高风险安全、数据迁移、分布式一致性和跨服务性能项进行 3–4 轮；
- 第一轮查正确性与安全，第二轮查复杂度与资源，第三轮查契约/数据迁移/降级，第四轮只在仍有新发现时进行；
- 连续一轮三条审查线都无新问题，且验证通过，才可判定收敛；
- 达到 4 轮仍不收敛时停止扩大修改，形成遗留清单交用户决策。

## 7. 前端专项设计边界

### 7.1 可直接实施的客观要求

- 视口至少覆盖 375、768、1024、1440 px；
- 不出现非业务需要的页面级横向滚动；
- 主导航在窄屏可用，不依赖固定 220px 侧栏；
- 交互元素支持键盘聚焦和 Enter/Space 激活；
- 图标按钮具有可访问名称；
- 页面包含 loading、empty、error、retry 和成功反馈；
- 表格和大列表有分页、滚动或虚拟化策略；
- `html lang`、标题、焦点可见性和颜色对比满足基本无障碍要求；
- 设计令牌和共享组件覆盖四类角色，避免同语义组件各自硬编码。

### 7.2 必须用户确认的视觉决策

- 是否延续玻璃拟态；
- 医生、药房、管理员是否使用同一视觉语言或更高信息密度变体；
- 品牌色、渐变、卡片透明度、圆角和阴影强度；
- 工作台布局、导航模式和移动端信息层级；
- 视觉回归基线截图。

实施时先用 2–3 个代表页面制作原型或对比，不直接批量修改 46 个页面。

## 8. 后端性能专项方法

每个性能任务按以下格式执行：

1. 明确输入规模变量，例如用户数 U、处方明细 N、药品种类 D、排班数 S；
2. 写出当前时间、空间和外部 I/O 复杂度；
3. 用日志、datasource-proxy/p6spy、Redis/Feign 指标或 SQL 计划验证静态判断；
4. 先优化 I/O 次数和数据规模，再考虑微观代码优化；
5. 对并发写链路同时验证正确性、幂等和补偿，不只看响应时间；
6. 优化前后使用同一数据集、并发度和环境；
7. 结果记录 P50/P95/P99、吞吐、错误率、SQL/远程调用数和资源峰值。

禁止仅凭以下信号宣称优化完成：

- 单元测试通过；
- 代码从循环改成 Stream；
- 新增缓存但未证明命中率和失效正确性；
- 新增索引但未提供 `EXPLAIN ANALYZE`；
- 异步化接口变快但没有证明最终一致性和失败恢复。

## 9. 契约与文档闭环

长期维护按业务域建立契约矩阵，至少包含：

| 字段 | 内容 |
| --- | --- |
| 业务域/接口 | docs 中的方法和路径 |
| 前端调用 | API 文件与调用页面 |
| 后端实现 | Controller、DTO、Service |
| 数据落点 | 表、字段和状态机 |
| 权限 | 角色、scope、owner/dataScope |
| 测试 | 单测、FlowTest、E2E、接口实调 |
| 一致性 | 一致、缺漏、矛盾、待决策 |
| action | 改前端、改后端、补 docs 或设置决策门禁 |

AI 域优先补齐 session/history、文件上传、反馈回复、影像列表、report-analysis 和 SSE 等已实现契约。内部 `/internal/**` 与对外 `/api/v1/**` 必须分开记录，避免把服务接口错误暴露给前端。矩阵必须显式标出 `image-detection`/`imaging-detection` 路径冲突、internal 用户 JWT 透传与 `requireService()` 冲突、患者签到前后端角色冲突、通用 payment patch 伪退款、处方前端缺口。

## 10. 质量门禁与退出条件

### 10.1 阶段门禁

- **文档门禁**：状态变更达到 E2，跨文档无新矛盾；
- **前端门禁**：构建成功，相关 Playwright 通过，console 无新增错误，关键视口无非预期溢出；
- **真实链路门禁**：mock-only E2E 不能作为上线门禁；关键角色旅程必须在真实后端联调下检查 console/network、错误恢复和权限拒绝；
- **后端门禁**：受影响模块编译/测试通过，接口契约一致；
- **性能门禁**：有可复现基线和对比，复杂度或 I/O 数量确实下降；
- **安全门禁**：正向、越权、未认证、跨 owner 四类用例覆盖；
- **数据门禁**：DDL、迁移、回滚/可重入策略和 `EXPLAIN` 证据齐全；
- **review 门禁**：至少两轮收敛，无未解释的高风险发现；
- **Git 门禁**：一项一 commit，工作区不夹带用户 WIP，不 push。

### 10.2 总体完成条件

8–12 周路线完成时应满足：

1. `修改建议.md` 逐条状态可由代码、DDL或测试复核；
2. 文档、前端、后端契约矩阵无未分类差异；
3. P0/P1 项已完成、明确降级或获得产品决策；
4. 关键角色旅程有 Playwright 回归；
5. 后端高风险热点有基线、优化和回归数据；
6. 核心 FlowTest 可稳定运行；
7. CI 能执行构建、关键测试和 E2E；
8. 指标能观察慢请求、错误、外部依赖、锁和消息；
9. 前端四角色在响应式、无障碍和交互反馈上达到共同门禁；
10. 最后一轮三线 review 无新高/中风险问题，或遗留项已进入明确 backlog。

## 11. 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| docs 已过时导致误修 | 阶段 0 先校准，并要求 E2 证据 |
| 已有计划与现状重复 | 实施前逐任务状态核对，只执行未完成步骤 |
| H2 与 MySQL 方言差异掩盖业务测试 | 测试 schema 隔离或 Testcontainers MySQL；核心流程不依赖生产种子 SQL |
| 性能结论只有静态推断 | 阶段 1 先建设测量地基，E3 后才确定方案 |
| AI 单线任务跨度过大 | 按依赖分批，每批独立上下文、review 和 commit |
| 前端批量换肤引入回归 | 先设计令牌和代表页面原型，用户确认后分角色迁移 |
| 分布式异步化牺牲一致性 | 中间态、幂等、重试、死信、对账和故障注入同时验收 |
| 用户 WIP 被覆盖 | 所有 add/commit 精确指定文件；`MedicalRecordFlowTest.java` 持续隔离 |

## 12. 下一步

根据本设计另行编写 8–12 周实施计划。计划需要：

- 映射每个阶段的具体文件和任务；
- 复用并校准已有 legacy/security 计划；
- 为每个缺陷给出证据、风险、依赖、验证命令和退出条件；
- 明确每个阶段的 review 与 commit 检查点；
- 把前端视觉方向标记为唯一需要用户确认的实施门禁。
