# MedConsult-Assist 三分支整合说明

> 分支：`MedConsult-Assist`（基于 `backend/dev`，整合 `origin/frontend` + `origin/backend/ai-stack`）
> 日期：2026-07-11

## 一、整合结论

经对权威设计文档（`docs/架构设计文档.md`、`docs/接口文档.md`、`public/项目要求/接口文档.txt`、`docs/修改建议.md`）的交叉核验：

- **`backend/dev` 符合规范**：全部 controller 用复数 REST（`/api/v1/patients`、`/drugs`、`/doctors`、`/appointments`、`/medical-records`、`/notifications`），与所有设计文档一致。
- **`origin/frontend` 偏离规范**：`src/api/*.js` 原用单数 + 动作式路径，已全部改写适配后端复数 REST。
- **`origin/backend/ai-stack` 的 `ai/` 模块**：依赖自己的扁平 `common` 模块（SB 3.3.7），已适配到 `backend/dev` 的 `medconsult-common`（7 子模块，SB 3.2.0）。

## 三、AI 服务整合要点

### 3.1 common 类适配

| ai-stack 类 | backend/dev 替代 | 说明 |
|---|---|---|
| `ApiResponse` | `Result`（common-core） | `ApiResponse.success(x)` → `Result.ok(x)` |
| `BizException(int,msg)` | `BusinessException(ErrorCode,msg)` | int code → ErrorCode 枚举映射 |
| `PageResult`(long) | `PageResult`(int)（common-core） | `new PageResult(...)` → `PageResult.of(...)` |
| `RedisRateLimiter` | `RateLimiter` 接口（common-redis） | 注入类型改接口 |
| `SseEventBus`(ai-stack) | `AiSseEventBus` 适配器（新建） | 桥接 SseEmitter 生命周期 + Redis pub/sub |
| `RequestContext`(web) | `AiHeaders` 常量类（新建）+ MDC traceId | header 常量本地化 |

### 3.2 鉴权模型统一

采用 **JWT 服务身份模型**（backend/dev 的 `SecurityContext.requireService()`）：
- AI 的 `/internal/ai/**` 每个方法首行 `SecurityContext.requireService()`
- AI 调用 patient/drug/medical-record 用 common-feign 客户端（`AuthRelayInterceptor` 透传服务 JWT）
- 删除了 AI 自己的 `InternalApiKeyFilter` / `AuthInternalClient` / `FeignInternalConfig`
- 新建 `AiServiceTokenProvider`：用 serviceCode + apiKey 调 auth-service 换发服务 JWT，缓存 Redis

### 3.3 auth-service 新增

- `sys_service_account` 表（种子数据：ai-service / dev-ai-service-api-key）
- `POST /internal/auth/service-token`：serviceCode + apiKey → SERVICE 类型 JWT

### 3.4 MQ 复用

AI 删除了自己的 `AiMqConfig`（DirectExchange 冲突），改用 common-mq 的 `MqConstants`：
- 影像检测任务：`EXCHANGE_AI_IMAGING` + `RK_AI_IMAGE_DETECT` + `QUEUE_AI_IMAGE_DETECT`
- 调用日志：`EXCHANGE_LOG` + `RK_AI_CALL_LOG` + `QUEUE_AI_CALL_LOG`

### 3.5 接口路径对齐前端

AI controller 路径已对齐前端 `src/api/ai.js`：
- `image-detection` → `imaging-detection`（submit/detail/review/list）
- `medical-record-summary` → `summary/by-record/{recordNo}` / `summary/by-text`
- `summary/confirm` → `PUT /ai/summary/{summaryId}/confirm`
- `call-logs` → `call-log`
- 新增 `symptom-chat/session`、`symptom-chat/history/{sessionId}`、`feedback/{id}/reply`

## 六、如何运行

### 6.1 后端
```bash
# 1. 启动中间件（Nacos / MySQL / Redis / RabbitMQ）
cd infra && docker-compose up -d
# Nacos 独立启动：D:\springcloudalibaba\nacos-server-2.3.0\nacos\bin\startup.cmd -m standalone

# 2. 编译全部模块（在 backend/ 目录下）
cd backend && ./mvnw clean compile

# 3. 打包并启动服务（在 backend/ 目录下）
./mvnw clean package -DskipTests
java -jar auth-service/target/auth-service-0.1.0-SNAPSHOT.jar
java -jar gateway/target/gateway-0.1.0-SNAPSHOT.jar
java -jar ai-service/target/ai-service-0.1.0-SNAPSHOT.jar
# ... 其他服务
```

### 6.2 前端
```bash
cd frontend
npm install
npm run dev    # http://localhost:3000（代理 /api → localhost:8080 网关）
```

### 6.3 AI 服务依赖的外部中间件
AI 服务完整运行还需：**MongoDB**（疾病知识库）、**Milvus**（向量检索）、**MinIO**（影像存储）、**DeepSeek API**（LLM，OpenAI 兼容）。

**详细部署步骤见 [AI服务外部依赖部署指南.md](./AI服务外部依赖部署指南.md)**（Windows / PowerShell 分步操作 + 验证命令）。

要点速览：
- MongoDB：本机安装（已就绪，监听 27017）
- Milvus 2.4 + MinIO：已写进 `infra/docker-compose.yml`，`docker compose up -d` 一键拉起
- DeepSeek：在 https://platform.deepseek.com 申请 key，写入 `infra/.env` 的 `DEEPSEEK_API_KEY`
- Embedding：DeepSeek 不提供 embedding 接口，需另配 OpenAI 或兼容服务（详见部署指南 §6）

### 7.2 运行时端到端链路验证（真实 HTTP 请求）
启动中间件（Nacos + Docker MySQL/Redis/RabbitMQ/Qdrant）+ gateway + auth-service + ai-service 后：

| 测试 | 路径 | 结果 |
|---|---|---|
| 用户注册 | `POST /api/v1/auth/register`（经 gateway） | ✅ code:0，用户落库 |
| 用户登录 | `POST /api/v1/auth/login`（经 gateway） | ✅ code:0，返回 accessToken/refreshToken（JWT） |
| 当前用户 | `GET /api/v1/auth/me`（带 JWT） | ✅ code:0，返回用户信息（role=PATIENT） |
| AI 调用日志 | `GET /api/v1/ai/call-log`（带 JWT，经 gateway） | ✅ code:0，返回 PageResult 结构 |
| AI 反馈提交 | `POST /api/v1/ai/feedback`（带 JWT，经 gateway） | ✅ code:0，feedbackNo 落库 |

**链路打通证据**：
- `frontend → gateway(8080) → auth-service(8081)`：register/login/me 全通
- `gateway → ai-service(8086)`：call-log/feedback 全通
- `ai-service → auth-service`：服务身份 JWT 换发通（AiServiceTokenProvider 调 AuthFeignClient）
- 三个服务均注册到 Nacos（DEFAULT_GROUP）

### 7.3 运行时验证发现并修复的问题
1. **spring-cloud-starter-loadbalancer 缺失**：ai-service 用 Feign 调其他服务，OpenFeign 4.1.0 不再传递 loadbalancer → 运行时 "No Feign Client for loadBalancing"。已补依赖。
2. **common-feign 新类未入 jar**：新建的 AuthFeignClient/MedicalRecordFeignClient 需 `clean install common-feign` 后重新打包 ai-service。
3. **LocalMessageMapper 未扫描**：common-mq 的 MessageDispatcher 依赖 LocalMessageMapper bean，ai-service 的 @MapperScan 需加 `com.medconsult.common.mq` 包。
4. **local_message 表缺失**：MessageDispatcher 扫描此表，ai-service schema 需加 local_message 表 DDL。
5. **MongoAutoConfiguration 冲突**：mongodb-driver-sync 5.1.2 与 Boot 3.2 MongoAutoConfiguration 的 StreamFactory API 不兼容，已排除该自动装配（ai-service 通过 MongoDiseaseRepository 手动建 MongoClient）。
6. **RabbitMQ 端口**：docker-compose 映射 5672:5672，ai-service 配置默认值从 5673 改为 5672。
7. **ai_feedback.admin_reply 列**：新增管理员回复字段，schema 加 `admin_reply`/`replied_at` 列。
8. **服务账号 BCrypt hash**：种子数据的 hash 修正为 dev-ai-service-api-key 的真实 BCrypt 摘要。

### 7.4 已知限制（不阻塞链路，需后续补齐）
