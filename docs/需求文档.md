# 智慧医疗系统需求文档

## 1. 项目概述

本系统面向医疗机构的门诊与临床辅助场景，建设一套包含传统医疗业务管理、患者/医护工作台与 AI 智能辅助能力的前后端系统。系统需要支持患者档案、医生排班、预约与退款、电子病历、处方审调剂、药品库存等基础业务闭环，同时提供症状自诊问答、智能分诊、病历摘要、用药提醒与禁忌分析、报告文本分析和医学影像初筛等 AI 能力。

## 2. 建设目标

1. 打通患者、医生、预约、病历、药品等核心业务数据。
2. 提供标准化 REST API，便于前端、移动端、第三方系统接入。
3. 引入 RAG、Function Calling、外部 AI 模型调用等能力，提升问诊、分诊、病历处理和用药安全效率；其中 RAG 只使用已导入的疾病 JSON 知识数据，不约束具体存储实现。
4. 对敏感医疗数据进行权限控制、审计记录和数据脱敏，满足医疗信息安全要求。

## 2.1 技术栈约束

本项目必须采用以下技术栈建设，后续详细设计、接口实现和部署方案需与本节保持一致。

| 层级 | 必选技术 | 使用说明 |
| --- | --- | --- |
| 基础底座 | JDK 21、Spring Boot 3.x | 使用 JDK 21 虚拟线程承载 AI 大模型长连接、SSE 流式输出和外部模型调用，降低 I/O 阻塞带来的线程资源压力。 |
| 微服务框架 | Spring Cloud Alibaba 轻量级方案 | 仅采用 Nacos 做注册与配置、Spring Cloud Gateway 做统一网关、OpenFeign 做服务间通信，不盲目堆砌其他组件。 |
| 持久层框架 | MyBatis-Plus、MySQL 8.0 | 传统业务和 AI 调用结果等结构化数据存入 MySQL 8.0，通过 MyBatis-Plus 快速构建数据层。 |
| 缓存与消息 | Redis、RabbitMQ | Redis 用于高频数据缓存、登录态辅助、验证码/限流和分布式锁；RabbitMQ 用于 AI 服务与业务服务的异步解耦。 |
| AI 与大模型 | Spring AI 或 LangChain4j | 用于智能分诊、病历摘要、用药分析、影像检测编排、SSE 流式输出和 Function Calling；症状自诊问答以本地 Embedding 模型向量化 + 规则引擎为核心，可叠加生成式 LLM 做口语化输入标准化与回答润色，回答须可追溯到疾病 JSON 命中片段。 |
| 向量数据库 | Milvus 或 Qdrant | 仅用于保存疾病 JSON 切分后的文本向量 Embedding，支撑 RAG 检索；疾病 JSON 仍是唯一知识来源。 |
| 前端与工程化 | Vue 3、Docker、GitLab CI/CD | 前后端分离，服务容器化部署，通过 GitLab CI/CD 完成构建、测试和发布流水线。 |

**技术使用边界**

1. RAG 的原始知识来源只能是疾病 JSON，Milvus 或 Qdrant 只保存该 JSON 派生出的向量索引。
2. AI 长耗时任务应优先通过虚拟线程、SSE、RabbitMQ 异步任务等方式处理，避免阻塞核心业务接口；症状自诊问答除外，该功能以向量检索为主。
3. Redis 可用于缓存科室、医生、排班可用号源、热点疾病检索结果，但不能作为核心医疗数据的唯一存储。
4. RabbitMQ 用于 AI 摘要、影像检测、通知投递、AI 调用日志等异步场景。

## 3. 用户角色

| 角色 | 说明 | 主要权限 |
| --- | --- | --- |
| 患者 | 就诊人或经授权的家属代理人 | 管理本人/被授权人档案、预约与退款、查看本人病历和处方、处方缴费、接收用药提醒、使用症状自诊 |
| 医生 | 接诊医生 | 查看分配就诊范围内的患者档案、管理病历、结构化开方并提交审方、查看排班、使用 AI 辅助工具并承担最终医疗判断责任 |
| 药房/库存管理员 | 审方药师与药品库存维护人员 | 审方、调剂、管理药品与批次、库存入库出库和预警、复核用药禁忌分析 |
| 医院管理员 | 系统运营人员 | 用户/医生/排班/科室配置、最小权限管理、审计查询、AI 调用与反馈治理；不默认获得无限患者数据读取权 |


## 4. 业务范围

### 4.1 传统业务域

#### 4.1.0 用户认证与账号管理

**需求说明**

系统需要提供基础的注册、登录和当前用户信息查询能力，为患者、医生、药房/库存管理员、医院管理员等角色提供统一身份认证入口。

**核心功能**

1. 用户注册。
2. 用户登录并获取访问令牌。
3. 刷新访问令牌。
4. 退出登录。
5. 查询当前登录用户信息。
6. 支持账号启用、禁用状态。

**关键字段**

用户编号、账号、手机号、密码、姓名、角色、关联患者编号、关联医生编号、账号状态、最后登录时间。

**业务规则**

1. 账号、手机号至少填写一项，且不能重复注册。
2. 密码存储必须使用不可逆加密摘要，不允许明文保存。
3. 登录成功后返回访问令牌和刷新令牌。
4. 退出登录后当前刷新令牌失效。
5. 禁用账号不可登录。
6. 医生账号可关联医生基础信息，患者账号可关联患者档案。

#### 4.1.1 患者档案管理

**需求说明**

患者档案用于记录患者基础信息、联系方式、过敏史、既往病史、家族病史和紧急联系人。系统应支持患者档案的创建、查询、修改、禁用和条件检索。

**核心功能**

1. 新增患者档案。
2. 查询患者档案详情。
3. 按姓名、手机号、证件号、患者编号分页检索。
4. 修改基础信息、病史信息、过敏史。
5. 标记患者档案状态，如正常、禁用、已合并。
6. 支持患者档案与预约、病历、用药记录关联。

**关键字段**

患者编号、姓名、性别、出生日期、证件类型、证件号、手机号、地址、过敏史、既往病史、家族病史、紧急联系人、档案状态。

**业务规则**

1. 证件号和手机号至少填写一项。
2. 同一证件类型和证件号不允许重复建档。
3. 删除档案建议采用逻辑禁用，不做物理删除。
4. 医生只能访问授权就诊范围内的患者档案。

#### 4.1.2 医生排班管理

**需求说明**

医生排班用于维护医生在不同科室、日期、时段的出诊安排，并作为预约挂号可用号源的基础数据。

**核心功能**

1. 新增医生排班。
2. 查询医生排班列表。
3. 按科室、医生、日期查询可预约号源。
4. 修改排班状态，如正常、停诊、已满。
5. 设置总号源数、已预约数和剩余号源数。

**关键字段**

排班编号、医生编号、医生姓名、科室编号、科室名称、出诊日期、时段、开始时间、结束时间、总号源、已预约数、挂号费、排班状态。

**业务规则**

1. 同一医生同一日期同一时间段不允许重复排班。
2. 停诊排班不可继续预约。
3. 已预约数不能大于总号源数。
4. 排班变更应通知已预约患者。

#### 4.1.3 预约挂号

**需求说明**

预约挂号用于患者选择科室、医生和排班号源完成预约，并支持取消、支付状态更新、签到和就诊状态流转。

**核心功能**

1. 创建预约订单。
2. 查询预约详情。
3. 按患者、医生、科室、日期、状态分页查询。
4. 取消预约。
5. 更新支付状态。
6. 签到、叫号、完成就诊。

**关键字段**

预约编号、患者编号、医生编号、科室编号、排班编号、预约日期、预约时段、挂号费、支付状态、预约状态、就诊序号、取消原因。

**业务规则**

1. 创建预约时必须校验排班可用、剩余号源充足。
2. 预约成功后应扣减剩余号源并生成就诊序号。
3. 取消预约后应释放号源。
4. 已完成就诊的预约不可取消。
5. 重复预约限制可按患者、科室、日期配置。
6. 已支付预约取消必须创建独立退款单，状态至少覆盖申请、处理中、成功和失败；不得只修改预约支付字段伪造退款成功。
7. 退款接口必须校验预约属于当前患者，并具备幂等、回调验签、对账和失败重试能力。

#### 4.1.4 电子病历存储

**需求说明**

电子病历用于保存医生接诊过程中的主诉、现病史、体格检查、诊断、医嘱和附件信息，并关联独立处方对象，作为 AI 摘要与用药分析的数据来源。

**核心功能**

1. 新增电子病历。
2. 查询病历详情。
3. 按患者、医生、时间范围分页查询。
4. 修改草稿病历。
5. 提交归档病历。
6. 关联预约、处方、检查报告、影像报告。

**关键字段**

病历编号、患者编号、医生编号、预约编号、主诉、现病史、既往史、体征、初步诊断、最终诊断、医嘱、附件、病历状态。处方通过 `record_id` 关联；若病历保留处方文本，只能作为不可变历史快照。

**业务规则**

1. 已归档病历不可直接修改，需走修订流程。
2. 病历访问必须记录审计日志。
3. AI 生成摘要只能作为辅助结果，不能替代医生诊断。
4. 病历中的敏感信息对非授权角色应脱敏。

#### 4.1.5 处方、审方与调剂

**需求说明**

处方是独立于病历的医疗业务对象，支持医生开立、药师审核、患者缴费、药师调剂和完成/退方的全链路追溯。用药分析以结构化处方及患者档案为输入，不直接解析可覆盖的病历自由文本。

**核心功能**

1. 医生为本人接诊范围内的病历创建结构化草稿处方。
2. 医生提交处方进入待审状态，药师记录通过、驳回和审核意见。
3. 患者查看本人处方详情并完成缴费。
4. 药师按 FEFO 选择批次调剂，记录处方明细与库存流水关系。
5. 完成处方或在允许的状态下退方，所有状态变化可审计。

**状态机**

`DRAFT -> PENDING_REVIEW -> APPROVED -> PAID -> DISPENSED -> COMPLETED`；审核可进入 `REJECTED`，允许的未完成状态可进入 `CANCELLED`。状态跳转必须由后端白名单校验，前端不得直接 patch 任意状态。

**关键字段**

处方编号、病历编号、患者编号、医生编号、科室编号、状态、审方药师、审方意见、总金额、支付状态、处方明细、药品与规格快照、剂量、频次、途径、天数、数量、单价、分配批次和已发数量。

**业务规则**

1. 医生只能为本人接诊/授权范围内的患者开方和提交。
2. 患者只能查看和支付本人处方，不能通过处方 ID 访问他人数据。
3. 药师审核与调剂职责分离于医生，审核结论和库存操作必须记录操作人。
4. AI 禁忌分析不能自动修改处方或覆盖药师审核，只能提供可追溯提醒。
5. 缴费、调剂、完成和退方必须幂等；调剂失败不得留下已扣库存但处方未推进的中间不一致。
6. 处方修改和取消保留历史版本或审计快照，不允许覆盖已完成医疗文书。

#### 4.1.6 药品库存管理

**需求说明**

药品库存用于维护药品基础信息、批次库存、入库、出库、库存预警和有效期预警，并为用药分析提供药品基础数据。

**核心功能**

1. 新增和维护药品信息。
2. 查询药品列表和详情。
3. 药品入库。
4. 药品出库。
5. 查询库存流水。
6. 库存不足和近效期预警。

**关键字段**

药品编号、通用名、商品名、规格、剂型、生产厂家、批准文号、批次号、库存数量、单位、最低库存阈值、有效期、禁忌信息、相互作用信息。

**业务规则**

1. 出库数量不能超过当前可用库存。
2. 批次库存应支持先进先出或按有效期优先出库。
3. 库存低于阈值时产生预警。
4. 过期药品不可出库。



### 4.2 AI 智能域

#### 4.2.0 AI 知识数据边界

**需求说明**

系统中的 RAG 检索只基于疾病 JSON 知识数据。文档只约束 JSON 内容结构和使用方式，不约束该 JSON 的存储介质或数据访问实现。

**JSON 参考字段**

疾病名称 `name`、疾病描述 `desc`、分类 `category`、预防方式 `prevent`、病因 `cause`、症状 `symptom`、医保状态 `yibao_status`、患病概率 `get_prob`、传播方式 `get_way`、并发症 `acompany`、推荐科室 `cure_department`、治疗方式 `cure_way`、治疗周期 `cure_lasttime`、治愈概率 `cured_prob`、费用说明 `cost_money`、检查项 `check`、推荐药物 `recommand_drug`、药物详情 `drug_detail`。

**业务规则**

1. 症状自诊问答和智能分诊的知识引用必须来自疾病 JSON。
2. RAG 引用结果需要返回命中的疾病名称、命中字段和片段内容。
3. 不要求接口暴露 JSON 的实际存储位置或底层数据访问细节。
4. 若疾病 JSON 中没有可用依据，AI 需要明确说明依据不足，不能编造疾病知识。

#### 4.2.1 症状自诊问答（RAG + 疾病 JSON 知识数据）

**需求说明**

患者输入症状、持续时间、年龄、性别等信息后，系统基于疾病 JSON 派生出的向量索引检索相关疾病资料，并根据检索结果返回健康咨询回答、可能原因、建议就诊科室和风险提示。该功能以本地 Embedding 模型向量召回 + 规则引擎为核心；当规则难以覆盖口语化输入或需自然语言润色时，允许叠加生成式 LLM 做输入标准化与回答重组，但回答内容必须可追溯到向量数据库命中的疾病 JSON 片段和系统规则，不得引入模型自由生成的医学论断。

**核心功能**

1. 接收患者症状描述并返回问答结果。
2. 基于疾病 JSON 中的 `name`、`desc`、`cause`、`symptom`、`acompany`、`cure_department`、`check` 等字段进行 RAG 检索。
3. 返回引用知识片段，便于结果解释。
4. 对高风险症状进行就医提醒。
5. 支持多轮问答上下文。
6. 返回向量检索命中的疾病、字段、片段和相似度分数。

**业务规则**

1. 回答必须包含“非诊断，仅供参考”的提示。
2. 对胸痛、呼吸困难、意识障碍、大出血等高危症状应返回急诊建议。
3. 生成结果需带疾病 JSON 引用来源。
4. 不允许直接给出确定诊断结论。
5. 不允许调用大模型补写疾病知识；若向量数据库无有效命中，应返回依据不足。

#### 4.2.2 智能分诊推荐

**需求说明**

根据患者症状、年龄、性别、持续时间、严重程度和既往病史，推荐适合挂号的科室，并给出推荐理由和优先级。

**核心功能**

1. 输入症状信息。
2. 返回推荐科室列表。
3. 返回推荐置信度和理由。
4. 标识是否建议急诊。
5. 可与预约挂号联动，查询推荐科室下的可用号源。

**业务规则**

1. 至少返回一个推荐科室。
2. 高风险症状优先推荐急诊。
3. 推荐结果应可追溯到输入症状和知识依据。

#### 4.2.3 病历自动摘要生成

**需求说明**

系统根据电子病历原文、诊疗记录、检查结果和处方生成结构化摘要，辅助医生快速回顾病情。

**核心功能**

1. 根据病历编号生成摘要。
2. 支持直接输入病历文本生成摘要。
3. 摘要包含主诉、诊断、治疗方案、用药、随访建议。
4. 支持医生确认后保存摘要。

**业务规则**

1. AI 摘要需标识生成时间和模型版本。
2. 摘要保存前应允许医生人工确认。
3. 若病历内容不足，应返回无法生成的原因。

#### 4.2.4 用药提醒与禁忌分析（Function Calling）

**需求说明**

系统基于患者档案、过敏史、病史、当前处方和系统内药品基础信息，调用药品查询、相互作用查询、禁忌规则查询等函数，生成用药提醒与风险分析。

**核心功能**

1. 分析处方药品是否存在过敏风险。
2. 分析药品之间是否存在相互作用。
3. 分析患者病史与药品禁忌。
4. 生成用药时间、剂量和注意事项提醒。
5. 返回 Function Calling 调用轨迹，便于审计。

**业务规则**

1. 高风险禁忌必须明确标红或返回高风险等级。
2. AI 分析不能自动修改医生处方，只能生成提醒。
3. 分析结果应可追溯到药品基础信息、患者档案、处方数据和调用函数。

#### 4.2.5 报告文本分析与医学影像初筛（调用外部 AI 模型）

**需求说明**

系统将“报告文本分析”和“原始医学影像初筛”作为两条独立流程。文本报告进入结构化文本分析接口；原始影像先上传受控对象存储，再异步调用视觉模型，返回异常类型、位置、置信度、风险等级和复核建议。两类结果分表保存并分别由医生复核。

**核心功能**

1. 提交报告文本，返回结构化异常摘要、风险等级和建议。
2. 上传原始影像并获取短期文件引用，提交影像检测任务后立即返回 `PENDING`。
3. 通过轮询或 SSE 接收进度、完成和错误事件。
4. 返回影像异常标签、位置/区域、置信度、风险等级和文字替代结果。
5. 记录外部模型调用状态、耗时、模型版本、文件引用和 owner。
6. 支持医生分别复核文本结果和影像结果，记录复核人、结论和时间。

**业务规则**

1. 外部 AI 模型返回结果仅作为初筛，不作为最终诊断。
2. 模型调用失败时应返回明确错误信息，并支持重试。
3. 涉及影像文件的访问 URL 必须使用短时 presigned URL 或等价受控访问，不返回永久公开直链。
4. 患者只可访问本人影像；医生只可访问接诊/授权范围内影像；模型调用使用最小必要数据。
5. Canvas/框选结果必须同时提供可读的文字异常列表，不能只靠图形或颜色表达。
6. 高风险结果应触发医生优先复核和明确通知，不能由模型自动写入最终诊断。

### 4.3 轻量补充业务

以下功能用于补齐系统基础闭环，不替代核心需求，也不引入复杂流程。

#### 4.3.1 科室与医生基础信息

**需求说明**

系统需要维护可挂号科室和医生基础信息，支撑医生排班、预约挂号和智能分诊推荐。

**核心功能**

1. 查询科室列表和科室详情。
2. 查询医生列表和医生详情。
3. 医生信息包含所属科室、职称、擅长方向、简介和启用状态。
4. 科室信息包含科室名称、介绍、位置和启用状态。

**业务规则**

1. 停用科室不可继续创建新排班。
2. 停用医生不可继续创建新排班。
3. 智能分诊推荐的科室应能与科室基础信息匹配。

#### 4.3.2 通知提醒

**需求说明**

系统需要为预约成功、预约取消、排班停诊、用药提醒等场景生成通知，便于患者和医生及时获知关键事项。

**核心功能**

1. 创建系统通知。
2. 查询用户通知列表。
3. 标记通知已读。
4. 支持预约、排班、用药提醒、AI 风险提示等通知类型。

**业务规则**

1. 通知只作为站内消息记录，不强制约束短信、邮件或第三方推送实现。
2. 预约取消、排班停诊、用药高风险提醒应生成通知。
3. 通知内容应避免展示完整身份证号、病历全文等敏感信息。

#### 4.3.3 附件文件记录

**需求说明**

系统需要记录病历附件、检查报告、影像报告等文件元数据，便于电子病历和影像异常检测关联使用。

**核心功能**

1. 创建附件记录。
2. 查询附件详情。
3. 按业务对象查询附件列表。
4. 支持病历、检查报告、影像报告等业务类型。

**业务规则**

1. 接口只记录文件元数据和访问地址，不约束文件实际存储方式。
2. 文件访问地址应由业务系统控制权限和有效期。
3. 附件删除建议采用逻辑删除。

#### 4.3.4 AI 结果反馈

**需求说明**

系统需要支持医生对 AI 摘要、用药分析、影像检测、分诊建议等结果进行反馈，用于后续优化和人工追溯。

**核心功能**

1. 对指定 AI 结果提交反馈。
2. 反馈内容包含是否有用、是否采纳、人工备注。
3. 查询 AI 结果反馈记录。

**业务规则**

1. AI 反馈不自动改变原始 AI 结果。
2. 医生确认结果应记录确认人和确认时间。
3. 反馈记录应纳入 AI 调用追溯范围。

## 5. 数据关联关系

1. 患者档案与预约挂号是一对多关系。
2. 医生排班与预约挂号是一对多关系。
3. 预约挂号与电子病历通常是一对一关系。
4. 电子病历可关联多个处方、检查报告、影像报告。
5. 药品库存为处方和用药分析提供药品数据。
6. AI 问答、分诊、摘要、用药分析、影像检测应记录调用日志，并可关联患者、医生、病历或预约。
7. 科室与医生基础信息为排班、预约和分诊推荐提供基础数据。
8. 附件记录可关联电子病历、检查报告和影像检测任务。
9. 通知记录可关联预约、排班、用药分析或 AI 风险提示。
10. 用户账号可关联患者档案或医生基础信息，用于权限控制和业务数据归属。
11. 已支付预约取消后关联独立退款单，退款单关联支付流水、回调和对账记录。

## 6. 非功能需求

### 6.1 安全与隐私

1. 所有接口默认要求身份认证。
2. 使用角色权限控制访问范围。
3. 患者身份证号、手机号、病历内容等敏感数据应加密存储或脱敏展示。
4. 病历查询、导出、AI 分析等操作必须记录审计日志。
5. AI 服务调用不得向无权限第三方暴露患者隐私数据。
6. 家属代理访问必须有可撤销授权、授权范围和有效期，不能仅凭知道患者编号访问。
7. 病历、处方、影像和 AI 输入按最小必要原则传递；文件 URL 和服务 token 必须有有效期。
8. 明确数据保留、归档和删除策略；审计记录与医疗文书不得随普通用户清理脚本无痕删除。

### 6.2 性能要求

| 场景 | 要求 |
| --- | --- |
| 普通列表查询 | P95 响应时间不超过 500ms |
| 预约创建 | P95 响应时间不超过 800ms |
| 症状自诊问答 | P95 响应时间不超过 10s |
| 病历摘要生成 | P95 响应时间不超过 15s |
| 外部影像任务受理 | P95 响应时间不超过 800ms |
| 外部影像 AI 检测完成 | 在模型正常可用时 P95 不超过 30s，通过轮询或 SSE 异步通知 |

### 6.3 可用性要求

1. 核心业务接口可用性不低于 99.9%。
2. AI 服务异常不能影响传统业务主流程。
3. 外部 AI 模型调用应设置超时、重试和降级策略。

### 6.4 可追溯性

1. 预约、病历、库存变更需保留操作人、操作时间和变更内容。
2. AI 结果需记录模型名称、模型版本、提示词版本、知识来源类型和调用耗时。
3. Function Calling 需记录函数名、入参摘要、调用结果和风险等级。
4. 处方审核、缴费、调剂、退方和退款需记录状态前后值、操作人、幂等键和关联流水。
5. AI 反馈应进入坏案例回放、规则/知识库修订和离线评测闭环。

## 7. 验收标准

1. 用户可完成注册、登录、刷新令牌、退出登录和查询当前用户信息。
2. 完成患者档案、医生排班、预约挂号、电子病历、处方、退款和药品库存的增删改查或等价业务操作。
3. 预约挂号可根据排班号源完成创建、取消、状态流转。
4. 病历可归档，归档后不可直接修改。
5. 药品库存支持入库、出库、流水查询和预警。
6. 症状自诊问答返回 RAG 引用和安全提示。
7. 智能分诊返回推荐科室、置信度、理由和急诊标识。
8. 病历摘要可根据病历编号或文本生成，并支持保存确认版本。
9. 用药分析返回过敏、禁忌、相互作用和提醒信息。
10. 报告文本和原始影像使用独立接口/数据表；影像任务异步返回并支持医生复核和时效 URL。
11. 所有核心接口具备统一响应结构、错误码和权限控制。
12. 科室与医生基础信息可查询，并能支撑排班和分诊推荐。
13. 预约、停诊和高风险用药场景可生成通知并支持已读状态。
14. 病历和影像相关附件可记录元数据并与业务对象关联。
15. 医生可对 AI 结果提交反馈，反馈记录可追溯。
16. 医生可结构化开方并提交审方，药师可审核/调剂，患者只能查看和支付本人处方，八态流转和库存流水可追溯。
17. 已支付预约退款具备独立退款单、owner 校验、幂等、失败重试和对账，不以字段 patch 伪造成功。
18. 胸痛、呼吸困难、意识障碍、大出血等高危症状在患者端优先展示急诊升级行动，并支持人工接管。
