## 二、为什么前期 backend 实现时没考虑到、多轮 review 没发现

### 2.1 分类框架（4 类）

| 类别 | 含义 | 责任归属 |
| --- | --- | --- |
| **A 有意降级** | 代码有明确 TODO/注释说明「先 X，后续改 Y」，是有意识的技术债 | 可接受 |
| **B 半成品** | DDL/枚举/接口契约已建，但实现层未跟进，且**无 TODO 标记** | 流程缺陷 |
| **C 纯遗漏** | 文档、契约、代码三处都没提 | 严重 |
| **D 文档已列但 review 未对照** | `修改建议.md` 列了，但 5 轮 review 没把它当 checklist 逐条核对 | 流程缺陷 |

### 2.2 5 项归类

| # | A 有意降级 | B 半成品 | C 纯遗漏 | D 文档已列未对照 |
| --- | --- | --- | --- | --- |
| 1 RBAC | ✅（`scope = List.of("*")` 冒烟期标注 + DataSeeder Redis 注释说明） | — | — | ✅ |
| 2 症状规则表 | — | ✅（DDL 有、RiskRuleEngine 硬编码无 TODO 说明为什么不查库） | — | ✅ |
| 3 id_no | ✅（字段 + schema 双 TODO，明确等 common 加密模块） | — | — | ✅ |
| 4 退费 | ✅（docs 自分类 P2，docs 已声明「后续迭代」） | — | — | ✅ |
| 5 审计 MQ | — | ✅（消费端完整 + 生产端 AuditLogEvent 注释「未来」） | — | ✅ |

**关键发现：C 类（纯遗漏）为 0。** 没有一项是"彻底被忘掉"——全部要么是有意识降级（A），要么是半成品欠债（B），且全部在 `修改建议.md` 列出但 review 未逐条对照（D）。

### 2.3 根因：三条 review 基线的天然盲区

fullstack-review 的三条核查基线，对"架构半成品"类问题都有盲点：

| 基线 | 能发现 | 对 5 项的盲区 |
| --- | --- | --- |
| 前端调用 → 后端 API 契约 | 接口字段失配、空列、假成功占位 | 5 项都不影响"现有接口能跑通"，前端不调 RBAC 表也不调 audit producer |
| docs 契约 → 实现一致性 | 文档明示但代码缺失的功能 | 5 项在 docs 里要么标 P2（退费）要么以 TODO 形式存在，docs 没把它们写成"硬性要求" |
| git diff 改动核查 | 本次提交引入的回归 | 5 项都是"长期欠债"，不在任何一次 diff 里"突然出现" |

### 2.4 核心流程缺陷：修改建议.md 从未被当 checklist 使用

`修改建议.md` 是一份现成的 14 条问题总览表，但 fullstack-review 的 SKILL.md 第 0 步「扫描 docs/ 基线」在 1–4 轮被简化为"看需求文档 + 接口文档"，没有逐条核对修改建议表。**这是最高杠杆的流程改进点。**

### 2.5 各项合理性评估

| # | 前期未做的合理性 | 是否应现在补 |
| --- | --- | --- |
| 1 RBAC | 合理（冒烟期 `*` 兜底是标准降级） | 是（用户已确认 DB 优先 + `*` 兜底，灰度切换不引发 403 回归） |
| 2 症状规则表 | 不合理（既然写了 DDL，应至少给 Entity/Mapper + 硬编码 fallback） | 是（接库但保留硬编码 fallback，零行为漂移） |
| 3 id_no | 合理（等 common 加密模块） | **是（TODO 自己说"待 common 加密模块就绪"，现在就是那个时点）** |
| 4 退费 | 合理（P2，非核心闭环） | 是（用户确认做；前端已误报"原路退回"构成对客误导，必须修） |
| 5 审计 MQ | 不合理（消费端都做了，生产端空着等于功能 0%） | 是（根因是生产端完全空白，必须可靠投递） |

## 三、目标实施策略（用户已确认范围）

| # | 范围决策 | 实施要点 |
| --- | --- | --- |
| 1 RBAC | **DB 优先 + `*` 兜底** | 新建 4 表 + Entity/Mapper；AuthServiceImpl 查 sys_role_permission；**用户无权限记录时兜底 scope=`*`**（避免 403 回归）；DataSeeder 补 sys_user_role；PermissionAspect 不变 |
| 2 症状规则表 | **接库 + 硬编码 fallback** | 3 Entity + 3 Mapper；RiskRuleEngine 重构为查库 + 硬编码降级；**库为空时行为与现状完全一致**；SymptomChatService 改注入 |
| 3 id_no | **完整做** | 新建 common-crypto 模块（AES-256-GCM + 密钥从 Nacos）+ MyBatis TypeHandler；新增 `id_no_hash` 列支撑唯一性/模糊检索；数据迁移脚本把存量明文加密 |
| 4 退费 | **完整流程** | refund_order 表 + `POST /appointments/{id}/refund` + RefundRequest/RefundResponse DTO + RefundService（Redis 锁防重复退款）+ 前端入口 + 修正"原路退回"误导文案 |
| 5 审计 MQ | **可靠投递** | AuditLogEvent 移到 common-mq；4 个业务服务加 common-mq 依赖；每服务 local_message 表；AuditLogProducer 用 AFTER_COMMIT 投递（事务提交后才发，避免审计早于业务落库） |

### 3.1 横切原则（所有 5 项必须遵守）

1. **不留无标记半成品**：任何"先 X 后 Y"的实现必须在代码注释里写明状态（避免问题 2/5 那种 DDL 在、逻辑不在、还没 TODO 的代码卫生问题）。
2. **降级路径零行为漂移**：RBAC 的 `*` 兜底、规则表的硬编码 fallback，上线后行为必须与现状逐字节一致，再逐步收紧。
3. **数据迁移可重入**：id_no 加密的迁移脚本必须可重复执行（用 hash 列是否已填充判断）。
4. **每项独立 commit**：5 项各自内聚，不混合提交；commit message 用 Conventional Commits 中文描述。
5. **仅 commit 不 push**：严格遵守 AGENTS.md，未经用户明确指示不 push。
