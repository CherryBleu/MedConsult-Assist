## 0. 文档定位与锁定决策

本文档**不重辩**已经确认的决策，只接受结论并落地为可实现的架构。下列决策视为冻结，任何实现不得违背：

| # | 冻结决策 | 出处 |
| --- | --- | --- |

| 2 | **SCA 2022.0.0.0 与 Boot 3.2.0 无官方兼容矩阵**。缓解方式是 pom 手动排除冲突传递依赖 + Nacos 启动冒烟，**不升级版本**。兜底：移除 SCA BOM，直接用 nacos-client | AGENTS.md 红线 1 |

## 1. 服务拓扑与表归属

### 1.1 服务清单

按业务域拆分为 **7 个服务 + 1 网关**，每个服务独占自己的数据。

| 服务 | 职责一句话 | 独占的库 schema | 对外端口 | 内部端口 |
| --- | --- | --- | --- | --- |
| `gateway` | 统一入口、路由、限流、鉴权前置、JWT 解析透传 | — | 8080 | — |
| `auth-service` | 用户/角色/权限/服务账号、登录签发 JWT、Token 校验 | `medconsult_auth` | — | 8081 |
| `patient-service` | 患者档案（含脱敏、字段级加密） | `medconsult_patient` | — | 8082 |
| `outpatient-service` | 科室、医生、排班、预约、签到、叫号 | `medconsult_outpatient` | — | 8083 |
| `medical-record-service` | 病历、处方（主+明细）、附件元数据 | `medconsult_record` | — | 8084 |
| `drug-service` | 药品、批次库存、出入库流水、FEFO 选批、库存预警 | `medconsult_drug` | — | 8085 |
| `ai-service` | 全部 AI 能力（症状自诊/分诊/摘要/用药/影像）、向量库 | `medconsult_ai` | — | 8086 |
| `notification-service` | 通知、审计、登录日志异步落库 | `medconsult_notify` | — | 8087 |

> **端口仅为建议默认值**，最终由 Nacos 注册，端口可改；关键约束是业务端口**只在内网开放，不对外**。

### 1.2 表归属矩阵（数据所有权）

**核心原则：每张表只能归一个服务独占。别的服务要数据只能 Feign 调，禁止跨库 join、禁止直连他库。违反=架构红线。**

> AI 知识数据存储分两处，均由 ai-service 独占访问：`disease_json_knowledge`（疾病 JSON 元数据，存 MySQL 的 `medconsult_ai` schema）；`disease_vector_index`（JSON 切片后的向量，存 Milvus/Qdrant）。原始 JSON 是唯一知识来源，向量库只存检索索引。

#### auth-service（`medconsult_auth`）
- 传统：`sys_user`、`sys_role`、`sys_permission`、`sys_role_permission`、`sys_user_role`、`sys_service_account`
- 新增：`login_log`（《修改建议.md》§2.2）

#### patient-service（`medconsult_patient`）
- `patient`

#### outpatient-service（`medconsult_outpatient`）
- `department`、`doctor`、`doctor_schedule`、`appointment`

#### medical-record-service（`medconsult_record`）
- 传统：`medical_record`、`attachment`
- 新增：`prescription`、`prescription_item`（《修改建议.md》§2.1）

#### drug-service（`medconsult_drug`）
- `drug`、`drug_stock_batch`、`drug_stock_flow`

#### ai-service（`medconsult_ai`）
- 元数据：`disease_json_knowledge`
- 会话：`ai_chat_session`、`ai_chat_message`
- 结果：`ai_triage_result`、`ai_medical_summary`、`ai_medication_analysis`、`ai_report_text_analysis`、`ai_image_detection`、`ai_feedback`、`ai_call_log`
- 规则配置：`symptom_rule`、`high_risk_symptom_rule`、`negative_rule`（《修改建议.md》§3.1）
- 向量库（Milvus/Qdrant）：疾病 JSON 切片向量

#### notification-service（`medconsult_notify`）
- `notification`、`audit_log`

#### gateway
- 无业务表

```
                                        ▼                                        │

                                   (ai 取数据，全部 Feign)

                        ┌─────────────────────────────────────────┐
                        │      notification-service (8087)         │
                        │      通知 / audit_log / login_log        │
                        └────────────────▲────────────────────────┘
                                         │ RabbitMQ 异步
                  ┌──────────────────────┼──────────────────────────┐
                  │                      │                          │
        所有服务通过 MQ 投递：    所有写操作 → audit.log.queue   登录 → 直接入库
        notification.send.queue   (异步审计)                   或经 MQ
```

**无环设计（红线 10）**：图中 Feign 同步调用链路不存在环。

**症状自诊对外链路**：`患者客户端 -> Gateway POST /api/v1/ai/symptom-chat -> ai-service POST /ai/symptom-chat`。该路由属于 `/api/v1/*` 用户接口，必须经过 JWT 校验和患者权限校验；`/internal/*` 仍不对外暴露。调用前不检查预约、挂号或病历是否存在。

- `ai-service` 不会调用 `medical-record-service` 的"取处方再分析"接口。当 medical-record 触发用药分析时（药师审方），**必须把整张处方 + 患者上下文内联**通过 `/internal/ai/medication-analysis` 发给 ai-service，AI 收到 payload 后不再反向 Feign 取数据。
- ai-service → patient/drug/medical-record 的查询都是**只读 GET**，单向，不会形成与"写"路径的环。

### 1.4 数据获取边界（架构红线）

1. 每个服务只连自己的 schema，跨服务取数据**必须 Feign**。
2. Feign 仅允许调 `/internal/*`，禁止绕过 Gateway 内部接口直连 `/api/v1/*`。
3. 禁止在数据库层面跨 schema 做 join、视图、触发器。
4. 禁止服务 A 直接读服务 B 的表（哪怕同实例 MySQL）。

## 2. 服务间通信

### 2.1 双通道隔离

| 通道 | 前缀 | 鉴权 | 暴露面 |
| --- | --- | --- | --- |
| 对外 | `/api/v1/*` | 用户 JWT（Gateway 解析后透传 user-id/role/scope） | 经 Gateway 对外 |
| 对内 | `/internal/*` | 服务 API Key（Feign 拦截器自动注入） | 仅内网，Gateway 不路由 |

Gateway 配置规则：**只把 `/api/v1/*` 转发到后端**，对 `/internal/*` 直接 404。后端服务的 `/internal/*` 监听端口通过部署网络隔离保证外部不可达（详见 §10）。

### 2.2 同步 vs 异步选型规则

> **判断口诀**：被调方挂了，调用方业务**必须失败** → Feign（同步）；被调方挂了，调用方可以**继续/稍后重试** → MQ（异步）。

| 用 Feign 同步 | 用 RabbitMQ 异步 |
| --- | --- |
| ai-service 查患者档案做分析 | audit_log 落库 |
| ai-service 查药品禁忌/相互作用 | 通知投递 |
| outpatient-service 调 ai-service 拿分诊结果展示给患者 | 影像检测任务（长耗时） |
| medical-record-service 调 ai-service 生成病历摘要（流式） | AI 调用日志落库 |
| 所有服务调 auth-service 校验 Token（高频，可加 Redis 缓存） | 库存预警计算 |

### 2.4 双模鉴权 Feign 拦截器

服务间调用有两种身份来源，拦截器都要支持：

```java
// common-starter/feign 模块提供
public class AuthRelayInterceptor implements RequestInterceptor {
    @Override
    public void apply(RequestTemplate tpl) {
        RequestContext ctx = RequestContext.current();
        if (ctx.isUserTriggered()) {
            // 模式 A：用户触发的链路，透传用户 Token（医生开方→drug 审计挂在医生名下）
            tpl.header("Authorization", "Bearer " + ctx.getUserToken());
            tpl.header("X-User-Id", ctx.getUserId());
            tpl.header("X-User-Roles", ctx.getRolesCsv());
        } else {
            // 模式 B：自动触发/无用户上下文（定时任务、回调），用服务自身 JWT
            tpl.header("Authorization", "Bearer " + serviceTokenProvider.get());
            tpl.header("X-Caller-Service", ctx.getCallerService());
        }
        // 永远透传 traceId
        tpl.header("X-Trace-Id", ctx.getTraceId());
    }
}
```

被调服务的 `/internal/*` 过滤器：
1. 若带用户 Token → 当作用户链路，鉴权按用户角色 + scope。
2. 若仅带服务 Token → 当作服务链路，鉴权按 `sys_service_account.scope`。
3. 二者皆无 → 401。

### 2.5 无循环依赖约束

详见 §1.3 红线 10。补充约束：
- ai-service 不允许持有任何**写**类的 Feign 客户端（medical-record/drug/outpatient 的写接口）。
- ai-service → 业务服务的 GET 调用，参数必须是**调用方传入的 ID**，不得基于自身表里的 `record_id` 反向触发新的业务动作。

```java
}

public interface RateLimiter {
    // Redis 滑动窗口，多实例共享计数
    boolean acquire(String key, int max, Duration window);
}

public interface SseEventBus {
    // §9.2 详解：多实例 SSE 广播
    void publish(String channel, Object event);
    void subscribe(String channel, Consumer<Object> handler);
}
```

## 4. 鉴权与权限（RBAC 落地）

### 4.1 双 Token 体系

| Token | 用途 | 存储 | 失效 |
| --- | --- | --- | --- |
| access_token | 用户访问 API | 客户端内存 | 短期（2h） |
| refresh_token | 续签 access_token | 客户端 + Redis 共享黑名单 | 长期（7d），登出/改密入黑名单 |

**关键**：access_token 无状态（多实例天然支持），refresh_token 黑名单 + 登出态放 Redis 共享，多实例一致。

### 4.2 双模 JWT（用户身份 + 服务身份）

- 用户 JWT：claim 含 `userId / roles[] / primaryRole / scope[]`。
- 服务 JWT：claim 含 `serviceCode / scope[]`（如 `ai-service` → `patient:read,drug:read,medical-record:read`）。

服务 JWT 由 auth-service 在服务启动时签发（服务用自身 API Key 换 JWT，缓存到本地，过期前自动续签），落地为 `sys_service_account`（见《修改建议.md》§2.3 表 6）。

### 4.3 权限矩阵落地

权限矩阵见《修改建议.md》§2.3。落地方式：

1. **接口标权限**：每个对外接口方法注解 `@Permission(code="prescription:review", dataScope="ALL")`。
2. **数据范围注入**：`@Permission` 的 `dataScope` 由 `PermissionAspect` 转成查询条件：
   - `ASSIGNED`：医生查病历时自动追加 `doctor_id = currentUser.doctorId`。
   - `DEPT`：科室管理员追加分科室条件。
   - `SELF`：患者查自身数据 `patient_id = currentUser.patientId`。
3. **角色权限缓存**：`role:{roleId}:permissions` 放 Redis，TTL 30min，权限变更时主动 invalidate。
4. **症状自诊授权**：`POST /api/v1/ai/symptom-chat` 标注 `@Permission(code="ai:symptom-chat", dataScope="SELF")`，仅 `PATIENT` 角色拥有该权限。患者身份从已校验 JWT 对应账号的 `patient_id` 注入，禁止依据请求体切换患者；账号未关联患者档案时返回业务错误并提示先建档。

### 4.4 Gateway 信任边界（关键假设）

```
[客户端] ──Bearer JWT──▶ [Gateway: 解析 JWT，写入 X-User-Id/X-User-Roles/X-Trace-Id]
                              │
                              ▼
                       [backend /api/v1/*]
```

**假设：业务服务的 `/api/v1/*` 端口在内网，外部不可直连。** Gateway 写入的 `X-User-*` 头在后端被信任。一旦这个网络隔离假设被打破，攻击者可伪造 `X-User-Id` 越权。

> **部署必须满足**（详见 §10）：业务容器端口仅在内部网络/Service Mesh 内可达，Gateway 是唯一出口。此假设需在部署文档和运维 checklist 中**显式声明**。

`/internal/*` 不接受 `X-User-*` 头来源 = Gateway；它走服务 JWT + `AuthRelayInterceptor` 自己注入。

**Token 过期信任模型**（重要）：`/api/v1/*` 业务服务的 AuthFilter **必须用 `JwtCodec.parse` 重新解析 Bearer JWT**（校验签名 + 过期），**不得**仅从 `X-User-*` 头重建 `JwtPayload` 跳过校验。原因：
1. Gateway 写 `X-User-*` 头仅为路由/限流便利，过期判定权威在签发方（auth-service）与解析方（业务服务）；
2. 若信任重建的头，Gateway 缓存或重放一个已过期 Token 会被后端无脑放行；
3. `PermissionAspect` 不重复校验 `exp`——它假定 `SecurityContext` 里的 payload 已经过 `JwtCodec.parse` 把关。

落地：业务服务的 AuthFilter 调 `jwtCodec.parse(token)` → 成功才 `SecurityContext.setPayload(payload)`。`JwtPayload.exp` 仅供审计/日志，不作放行依据。

## 5. AI 域架构

### 5.1 症状自诊（不调生成式 LLM）

完全遵循《修改建议.md》§3.1。处理链路（**全在 ai-service 内**，不 Feign）：

该能力是患者登录后的直接可用功能，不依赖预约、挂号、接诊或病历。Gateway 对外暴露 `POST /api/v1/ai/symptom-chat`，校验 Bearer JWT、`PATIENT` 角色和 `ai:symptom-chat` 权限后，将可信用户身份透传至 ai-service；ai-service 以登录账号绑定的 `patient_id` 创建或续接本人会话。

```
POST /api/v1/ai/symptom-chat（Gateway 对外）
  │ 路由到 ai-service POST /ai/symptom-chat
  │
  ① 校验登录患者身份，从 JWT 上下文取得 patient_id（忽略或校验请求体 patientId）
  ② 轻量症状抽取层（基于 symptom_rule 表的规则匹配，非 LLM）
     "我心慌得厉害三天了" → {症状:"心悸", 持续时间:"3天", 严重程度:"明显"}
  ③ 本地 Embedding 向量化（bge-m3，离线，不联网）
  ④ 向量召回 topK（Milvus/Qdrant，按 disease_json 字段切片）
  ⑤ 规则过滤：negative_rule 剔除否定（"我没发烧"剔除"发烧"）
  ⑥ 规则判定：high_risk_symptom_rule 命中 → CRITICAL 强制提权
  ⑦ 模板拼装 answer + citations + suggestedDepartments
  ⑧ 写 ai_chat_message，记录 patient_id / query_embedding_model / rule_version
```

**不流式**（非生成式），同步返回。性能要求 P95 < 10s（embedding+检索为主耗时）。

### 5.2 影像检测（异步 + 文件上传）

文本报告分析与影像图像检测是**两个独立接口**（《修改建议.md》§3.2）：

- `POST /ai/report-analysis` → 文本 NLP，可同步或短异步
- `POST /ai/image-detection` → 图像视觉，**强制异步**：
  ```
  POST /ai/image-detection → 立即返回 {detectionId, status: PENDING}
  GET  /ai/image-detection/{id} → 轮询状态
  GET  /ai/image-detection/{id}/stream → SSE 推送完成事件（可选）
  ```

文件上传（ DICOM 几十 MB）走 medical-record-service 的 `POST /files/upload`（分片支持），生成 OSS/MinIO URL 后传给 ai-service。

### 5.3 SSE + Spring AI Flux 转换模式

病历摘要、用药分析、分诊（**症状自诊不流式**）提供 SSE 流式版本。Spring AI 返回 `Flux<String>`，转 `SseEmitter`：

```java
@GetMapping(value = "/ai/medical-record-summary/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter streamSummary(@RequestParam Long recordId) {
    SseEmitter emitter = new SseEmitter(0L); // 不超时（虚拟线程承载长连接）
    // 从 RequestContext 取调用方身份，调 /internal 拿病历全文
    String recordText = recordClient.getFull(recordId);
    chatClient.prompt().user(buildSummaryPrompt(recordText)).stream()
        .subscribe(
            token -> emitter.send(SseEmitter.event().data(token)),
            err -> emitter.completeWithError(err),
            () -> emitter.send(SseEmitter.event().data("[DONE]").name("done"))
        );
    return emitter;
}
```

**多实例下 SSE 连接的实例亲和性问题见 §9.2。**

### 5.4 Function Calling（用药分析）

用药分析走 Spring AI Function Calling，函数 schema 见《修改建议.md》§4.4。关键：

- Function 的数据源**禁止**让 AI 自由调外部接口；统一封装为受控的内部 Function bean：
  - `queryPatientAllergies(patientId)` → patient-service `/internal/patients/{id}/allergies`
  - `queryCurrentMedications(patientId)` → medical-record-service 查 `prescription_item` where prescription.status != CANCELLED
  - `queryDrugContraindications(drugId)` → drug-service `/internal/drugs/{id}/risk-info`
  - `queryDrugInteractions(drugAId, drugBId)` → 同上
- 所有 Function 调用走 Feign（用户或服务身份），结果入 `function_trace`。

### 5.5 ai-service 数据获取边界

- 只允许**只读 GET** Feign 到 patient/drug/medical-record。
- 上述 Function 的查询参数必须是**本次请求 payload 里的 ID**，不允许 AI 自主决定查任意患者（防越权）。
- 严禁 ai-service 持有业务写接口的 Feign 客户端。

### 5.6 ai_call_log 补充字段

落地《修改建议.md》§2.4 建议 5：`caller_service / trigger_user_id / trace_id / cost_tokens / request_id`。`request_id` 作幂等键。

## 6. 跨服务数据一致性

### 6.1 本地消息表模式（最终一致）

写业务 + 写本地消息表在**同一本地事务**，事务提交后由扫描器投递 MQ，消费者幂等。

```
[medical-record-service]
   Tx1: INSERT prescription
   Tx1: INSERT local_message(prescription_created, payload)
   ───commit───▶
   Scheduler 扫 PENDING 消息 → publish RabbitMQ → status=SENT
   收到 ACK → status=CONFIRMED；NACK/超时 → 退避重试，retryCount++，达上限进 DLQ

[notification-service consumer]
   consume prescription_created → 写通知
   幂等：以 local_message.messageNo 为去重键，Redis SETNX + 业务唯一索引双保险
```

**不做 2PC、不引入 Seata**（红线 1）。这意味着跨服务写是**最终一致**，开发者必须：
1. 写中间态供观测（如处方 `DRAFT→PENDING_REVIEW→APPROVED→PAID→DISPENSED→COMPLETED`）。
2. 消费失败要有补偿/人工兜底入口（DLQ + 监控告警）。

### 6.2 处方状态机（跨服务）

```
DRAFT ──submit──▶ PENDING_REVIEW ──approve──▶ APPROVED ──pay──▶ PAID
                       │                          │
                     reject                   cancel
                       ▼                          ▼
                   REJECTED                  CANCELLED

APPROVED/PAID ──dispense──▶ DISPENSED ──complete──▶ COMPLETED
```

跨服务流转点：
- `medical-record` 持有 prescription 表，状态机由此服务主导。
- 调剂发药 `dispense` → medical-record 通过本地消息表通知 drug-service 做 FEFO 出库，**不阻塞**返回。
- 出库成功后 drug-service 回写 `prescription_item.allocated_batch_id` + `dispensed_quantity`（通过反向消息或同步 Feign，二选一，建议反向消息避免环）。

> **⚠️ 实现现状（2026-07 batch 2）：dispense 暂用同步 Feign，非上述 MQ 异步。**
> 务实决策：drug-service 当前是纯同步服务（零 MQ 消费者，pom 不依赖 common-mq），
> 若走 MQ 需 drug-service 从零接入 common-mq（pom + yml + 消费者），工作量翻倍且无范例。
> 故 batch 2 采用 **同步 Feign**：medical-record `dispense` 直接调 drug-service
> `POST /internal/drugs/{drugNo}/outbound`（FEFO 扣减），立即拿结果反馈给药师。
>
> **同步实现的已知限制**：
> 1. 跨服务事务不一致——drug-service 的 outbound 是独立事务已提交，medical-record 事务回滚
>    无法回滚 drug 那边扣减。失败时处方保持原状态 + 日志供人工对账。
> 2. dispense 阻塞（等 drug-service 返回），非"不阻塞"。
>
> **MQ 异步化作为后续优化待办**：当 notification-service 落地 common-mq 后，可将 dispense
> 改为本地消息表 + MQ 模式（medical-record 写 DISPENSING 中间态 → MQ → drug-service 消费出库 →
> 反向消息回写 allocated_batch_id），彻底解决事务一致性。届时恢复本节描述的 MQ 流程。

### 6.3 多实例扫描去重

本地消息表扫描器在**每个实例**都跑，不去重发送端：
- 重复发送由**消费者幂等**吸收（`messageNo` 唯一键）。
- 不引入 ShedLock 给发送端选主——更简单，且消费者幂等本就是必须项。

> 与 §9.5 的定时任务选主区分：**业务定时任务**（库存预警、近效期、过号）用 ShedLock 选主避免重复计算；**消息投递扫描**靠消费者幂等。

## 7. 基础设施约定

### 7.1 Redis 键命名与用途

| 场景 | Key 模式 | 类型 | TTL |
| --- | --- | --- | --- |
| access_token 黑名单 | `jwt:black:{jti}` | string | =剩余有效期 |
| refresh_token 状态 | `auth:refresh:{userId}` | hash | 7d |
| 角色权限缓存 | `role:{roleId}:perms` | set | 30min |
| 号源配额（抢号原子扣减） | `schedule:{id}:quota` | string | 到就诊日结束 |
| 药品库存锁 | `lock:drug:{drugId}:stock` | string + Lua | 30s 租约 |
| 支付幂等锁 | `lock:payment:{paymentNo}` | string | 10min |
| AI 接口限流 | `ratelimit:ai:{userId}` | zset 滑动窗口 | 滚动 |
| 可预约号源缓存 | `schedule:available:{date}` | hash | 5min |
| 病历热点 | `medical:record:{id}` | string | 10min |
| SSE 完成事件广播 | `pubsub:sse:{userId}` | Pub/Sub | — |

**禁止**用本地 Caffeine 做业务并发/缓存一致性场景；Caffeine 仅限只读字典（科室列表等）且容忍秒级不一致。

### 7.2 分布式锁/限流统一封装

由 `common-redis` 提供（§3.2）。**业务并发禁止用 JVM 锁**（多实例下失效）。所有锁/限流必须走 Redis。

### 7.3 Nacos 配置约定

```
# 命名空间
medconsult-dev / medconsult-test / medconsult-prod

# 配置分组
group=COMMON          # 公共：数据源、Redis、MQ、JWT 密钥、虚拟线程开关
group=GATEWAY         # 路由规则、限流策略
group=AUTH            # Token 时效、密码策略
group=AI              # 模型名、外部服务商、API Key、限流阈值

# 共享配置
medconsult-common.yaml   # 所有服务共享（数据源模板、Redis、MQ、日志、虚拟线程）
medconsult-{service}.yaml # 各服务私有
```

**敏感配置**（数据库密码、JWT 密钥、外部 AI API Key）：Nacos 配置 + 加密，密钥由部署期 KMS/环境变量注入，不入仓库。

### 7.4 traceId 透传

- Gateway 入口生成 `X-Trace-Id`（无则生成）。
- Feign 拦截器透传。
- RabbitMQ 消息 `headers.traceId` 透传（生产者写，消费者入 MDC）。
- 日志格式含 `[%X{traceId}]`，全链路可串。

### 7.5 审计与登录日志异步落库

- 所有写操作 → AOP `@AuditLog` 拦截 → 发 `log.exchange` → `audit.log.queue` → notification-service 异步落 `audit_log`。
- 登录 → 直接写 `login_log`（auth-service 自己的库，同步写，登录路径不长）+ 发审计事件。

**风险**：SCA 2022.0.0.0 官方针对 Spring Boot 3.0.x，与 3.2.0 无兼容矩阵。可能冲突的传递依赖：`spring-core`、`spring-context`、`spring-web`、`spring-cloud-commons`、`spring-cloud-loadbalancer`。

**处理策略**：

1. **BOM 顺序**：`spring-boot-dependencies` 在前，`spring-cloud-dependencies`（2022.0.x）次之，`spring-cloud-alibaba-dependencies`（2022.0.0.0）最后，让 Boot/Spring Cloud 版本优先。
2. **手动排除冲突**：用 `<exclusions>` 移除 SCA 引入的低版本 spring-* 传递依赖。
3. **冒烟测试前置**：Nacos 注册+拉配置+一个 Feign 调用链路先跑通，再铺开业务功能。
4. **兜底方案**：若 SCA BOM 仍冲突到无法启动，**移除 `spring-cloud-alibaba-dependencies` BOM 导入**，改为单独引入 `spring-cloud-starter-alibaba-nacos-discovery` / `spring-cloud-starter-alibaba-nacos-config` 坐标并显式锁定其版本（不再让 SCA BOM 管理传递依赖版本）。这是既定决策，**不升级 Boot/SCA 版本**。

冒烟 checklist（写在每个服务的 README）：
- [ ] 应用启动无 `NoSuchMethodError` / `IncompatibleClassChangeError`
- [ ] Nacos 注册可见，配置拉取成功
- [ ] Feign 调用另一服务 200
- [ ] 虚拟线程开启（日志/actuator 验证）

### 8.3 虚拟线程全局开启

## 9. 多实例部署与水平扩展

> **默认假设：每个服务至少 2 副本**。所有并发控制按分布式语义设计，禁止任何"单实例假设"。

### 9.1 服务发现与负载均衡

- **Nacos 注册**：服务实例启动注册，下线主动注销 + 心跳超时被动摘除。
- **客户端负载均衡**：SCA 2022.0.0.0 用 Spring Cloud LoadBalancer（取代 Ribbon），Feign 默认集成。
- **健康检查**：`/actuator/health` 暴露给 Nacos 与 K8s readinessProbe。

### 9.2 SSE 长连接实例亲和性（最关键的坑）

**问题**：SSE 长连接绑在实例 A，AI 推理任务可能在实例 B 完成，结果推不到持有连接的实例 A。

**方案：Redis Pub/Sub 广播**（推荐默认，**不采用 Gateway 粘性路由**——粘性路由有单点风险，且扩缩容时连接重分布）。

```
[实例A] 持有 SseEmitter(userId=42)
   │ 订阅 Redis pubsub:sse:42
   │
[实例B] AI 任务完成
   │ publish pubsub:sse:42 → {event: done, data: ...}
   │
[Redis] 广播到所有订阅者
   │
[实例A] 收到 → 本地 emitter.send()
[实例B/C] 也收到但未持有该连接 → 忽略
```

封装在 `common-redis.SseEventBus`（§3.2）。每个实例：
- 建 SSE 连接时订阅 `pubsub:sse:{userId}`，`userId` 来自 JWT。
- AI 任务完成时 publish 到该 channel。
- 持有连接的实例本地推送，其他实例忽略。

**为什么不粘性路由**：
1. 粘性路由使某个用户始终绑在一个 Gateway/实例，扩缩容时连接重分布成本高。
2. 实例宕机时其上所有 SSE 连接丢失，无 fallback。
3. Redis Pub/Sub 是云原生标准做法，无亲和性约束，扩缩容自由。

**连接清理**：实例 shutdown 时，`graceful shutdown` 让在飞 token 推完，emitter.complete()；客户端断开后实例自动取消订阅。

### 9.3 状态/缓存共享

| 数据 | 多实例方案 |
| --- | --- |
| access_token（无状态） | 天然支持，多实例任意可验 |
| refresh_token / 登出态 | Redis 共享 |
| 角色权限缓存 | Redis 共享 |
| 分布式锁 | Redis 共享（§7.1） |
| 限流计数 | Redis 滑动窗口共享 |
| 业务热点缓存 | Redis 共享 |
| 只读字典（科室等） | Caffeine 本地可接受 |

### 9.4 文件存储

附件/影像**强制 OSS/MinIO**，**禁止**本地磁盘存储——多实例下本地文件不可见。

### 9.5 定时任务选主

业务定时任务（库存预警、近效期、过号处理）用 ShedLock（§8.4）保证多实例下只跑一份，避免重复计算/重复通知。

**例外**：本地消息表扫描器**不选主**，靠消费者幂等吸收重复发送（§6.3）。

### 9.6 优雅上下线

- **下线**：`server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase=30s`，让虚拟线程上的在飞请求/SSE token drain 完毕；同时主动调 Nacos 注销，避免新流量继续进来。
- **上线**：服务就绪（连上 Nacos、Redis、MQ）后才注册到 Nacos； readinessProbe 通过前不接流量。

### 9.7 容量画像与副本起点

| 服务 | 资源画像 | 副本起点 |
| --- | --- | --- |
| gateway | 轻计算、高并发、IO 密集 | 2+ |
| auth-service | 轻计算、高频 | 2 |
| patient-service | 轻计算 | 2 |
| outpatient-service | 中等（抢号锁） | 2 |
| medical-record-service | 中等 | 2 |
| drug-service | 中等（库存锁、FEFO） | 2 |
| **ai-service** | **重计算 + 长连接**（embedding/LLM/视觉） | 2+ **独立资源池** |
| notification-service | IO 密集（消费 MQ） | 2 |

ai-service 因长连接 + 重计算 + 外部模型调用，**独立资源池**，不与业务服务混部，便于单独扩缩容和故障隔离。

## 10. 部署与网络拓扑

### 10.1 容器化

每个服务一个镜像，多阶段构建（JDK 21 分层基础镜像）。`Dockerfile` 模板统一在 `medconsult-parent/build/` 下。

### 10.2 网络拓扑与隔离假设

```
                        ┌─────────────── 公网 ──────────────┐
                        │                                   │
                        ▼                                   │
              [Ingress / SLB]                               │
                        │                                   │
                        ▼                                   │
            ┌─────────────────────┐                        │
            │   gateway (8080)    │  ← 唯一对外出口          │
            └──────────┬──────────┘                        │
                       │ 内网                               │
        ┌──────────────┼──────────────┐                    │
        ▼              ▼              ▼                    │
   [业务服务集群 /api/v1/* 与 /internal/*]                 │
   auth/patient/outpatient/record/drug/ai/notification     │
        │                                                   │
        ├────▶ [Redis 集群]      （内网）                    │
        ├────▶ [RabbitMQ 集群]   （内网）                    │
        ├────▶ [MySQL]           （内网）                    │
        ├────▶ [Nacos 集群]      （内网）                    │
        └────▶ [OSS/MinIO]       （内网或专线）              │
```

**网络隔离硬约束（§4.4 假设的落地）**：
1. 业务服务容器端口**仅在内部网络可达**，外部只能通过 Gateway。
2. K8s NetworkPolicy / 安全组限制：只有 Gateway 容器能访问业务服务的 `/api/v1/*`；业务服务间可互通 `/internal/*`；外部网段拒绝直连业务端口。
3. `/internal/*` 端口对公网零暴露，Gateway 也不路由 `/internal/*`。

### 10.3 CI/CD（GitLab）

```
代码提交 → GitLab CI
  ├─ build:     mvn verify（先 medconsult-common install）
  ├─ test:      单测 + 集成测试（Testcontainers: MySQL/Redis/RabbitMQ）
  ├─ image:     构建镜像推 Harbor
  ├─ deploy-dev:  自动部署 dev
  └─ deploy-prod: 人工触发，蓝绿/滚动
```

**冒烟前置**（红线 1）：每次部署后跑 Nacos 启动 + Feign 链路冒烟，失败回滚。

```

     Result/异常/traceId/MyBatis 填充/JWT/@Permission/
     Feign 拦截器/分布式锁/限流/SSE EventBus/本地消息表
 └─ pom 冲突处理 + Nacos 启动冒烟（§8.2）

 ├─ auth-service（RBAC 五表、双模 JWT、登录日志）
 ├─ patient-service
 ├─ medical-record-service（含 prescription + prescription_item + 状态机）
 ├─ drug-service（含 FEFO 选批、current_stock）
 └─ notification-service（audit_log + 通知 + MQ 消费者）

第二阶段（P0 AI 与服务化）
 ├─ ai-service 症状自诊（规则链路 + 本地 Embedding + 3 张规则表）
 ├─ ai-service 影像检测拆两接口 + 文件上传
 ├─ 内部 Feign 接口契约全部落地（§2.3）
 └─ ai_call_log 补字段

第三阶段（P1 技术能力）
 ├─ SSE 流式接口（病历摘要/用药分析/分诊）+ Redis Pub/Sub 广播（§9.2）
 ├─ RabbitMQ 异步语义全部落地（影像/通知/审计/日志/库存预警）
 ├─ Redis 锁/限流实战（抢号/库存/支付幂等/AI 限流）
 ├─ Function Calling + drug 表结构化
 └─ ShedLock 定时任务（库存预警/近效期/过号）

第四阶段（P1/P2 收尾）
 ├─ 字段一致性统一（deleted/枚举/JSON 类型）
 ├─ 敏感数据加密（id_no AES、password BCrypt）
 ├─ 就诊闭环（签到/叫号/复诊关联）、退费、FEFO 多批混合出库
 ├─ 多实例部署验证（每个服务 2 副本，SSE/锁/限流/选主全验证）

```
