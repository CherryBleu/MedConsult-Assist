## 2. 应用场景

### 2.1 患者：诊前、诊中、诊后连续服务

1. 诊前：注册建档、查看医生与排班、预约缴费、症状问诊和智能分诊。
2. 到院：本人签到、查看候诊状态和叫号通知。
3. 诊中：授权医生查看本人档案、病历和必要影像；AI 只提供可追溯辅助信息。
4. 诊后：查看病历与处方、处方缴费、获取用药提示、申请符合规则的退款。

患者端必须优先展示急症升级建议、风险等级和下一步行动。AI 结果不得伪装成诊断，不得把引用证据或模型话术放在急症提示之前。

### 2.2 医生：接诊与临床决策辅助

1. 查看本人排班和分配给本人的预约，执行接诊状态流转。
2. 在授权范围内查看患者档案、历史病历和影像。
3. 书写结构化病历与处方，提交处方审方。
4. 使用病历摘要、用药分析、影像辅助；医生对最终医疗判断和病历内容负责。

当前医生病历已可书写，结构化开方和提交审方已有前端闭环；医生查看、列表、更新草稿和归档病历已按当前 JWT doctorId 限制到本人 `doctor_id`。`3729f0c` 已收紧病历 create 时的医生归属校验，请求医生必须解析为 JWT doctorId；主窗口验证 medical-record-service test BUILD SUCCESS，MedicalRecordFlowTest 47 tests passed。但 `appointment_no` 仍未反查患者/医生接诊归属，真实后端 E2E 不能外推为完成。

### 2.3 药师：审方、调剂和药品安全

1. 按状态处理待审处方，记录通过、驳回和意见。
2. 缴费后执行 FEFO 调剂，保证批次和处方明细可追溯。
3. 处理库存入库、出库、预警和流水。
4. AI 用药分析只提供辅助风险提示，不能覆盖药师的受控规则结果。

当前审方和调剂页面存在，调剂仍是同步跨服务调用；真实支付回调、失败恢复和对账仍需完善。

### 2.4 医院管理员：运营、权限、审计和 AI 治理

1. 管理用户、科室、医生、排班、药品和库存。
2. 管理角色与权限，查看审计日志，追踪关键写操作。
3. 查看 AI 调用、Token、成本、缓存、反馈和医生复审结果。
4. 只可访问职责所需的数据，不因管理员身份默认获得无限患者数据读取权。

RBAC 五表已建，login/refresh/internal user roles 在存在 RBAC 行时已优先读取 RBAC 事实源，无 RBAC 行仍保留 Redis/静态映射兜底；用户列表已在 `f5b6c30` 改为 RBAC 行存在时优先显示 RBAC 主角色。`sys_permission` 无 enabled 字段、全域权限矩阵和接口标注仍未闭环。审计消费端和 common-mq 生产端 AOP/outbox 已建，patient-service 患者档案写操作、outpatient-service 预约写操作、auth-service 登录/改密/补建档绑定、medical-record-service 病历/附件/处方关键写操作与 drug-service 药品/库存关键写操作已接入，ai-service 影像任务提交已通过事务 `local_message` outbox 入队，不能把空审计表视为“没有风险操作”。

### 3.1.1 开发启动基线

`.vscode/launch.json` 只维护可部署 Spring Boot 微服务，不配置 `medconsult-common` 或 `data`。每个启动项必须声明对应 Maven `projectName`、模块目录 `cwd` 和 `classPaths: ["$Runtime"]`，避免 Java 调试器混入 test scope。common 模块允许在 `src/test` 保留测试用 `@SpringBootApplication` 锚点，但不得提供 `main()`；`data` 是导入工具，有 `main()` 也不能作为生产服务挂到 Spring Boot Dashboard。当前启动配置基线来自 `64fdf89`、`013cf1d`、`8e4dd1e`：变更后运行 `python scripts/review/verify_vscode_launch.py`，校验 8 个可部署服务、禁止 common/data/envFile，并确认 common 测试锚点没有 `main()`。该脚本当前由 A2 维护，本文件只记录已提交口径，不代表本轮脚本改动已由 D2 复验。

VS Code / Spring Boot Dashboard 启动项必须使用各自模块目录作为 `cwd`，例如 `ai-service` 对应 `${workspaceFolder}/backend/ai-service`，避免多模块根目录下的相对路径、日志和资源解析歧义。2026-07-19 本地冒烟结果：`ai-service` 可在模块工作目录启动，Tomcat 监听 `8086`，以 `127.0.0.1:8086` 注册到 Nacos，RAG readiness 为 `ready=true`。该冒烟早于后续规则迁移和 `14bb3a2` RAG ranking 提交，不能外推为当前分支后续 8086 runtime smoke 已通过。

## 4. AI 问诊与治理基线

### 4.1 允许的能力

- 本地规则：高危症状、否定词、用药禁忌等确定性规则优先。
- 结构化输出：风险等级、急症建议、可能原因、建议科室和引用证据。
- RAG：Mongo 文档、Embedding、Milvus 向量检索及可追溯 citation。
- 受控工具：服务端白名单只读查询、参数 schema、次数/超时限制、审计 trace。
- 流式交互：SSE 进度、完成和错误事件。
- 多模态：报告文本和医学影像分开处理，医生复审最终结果。
- 治理：调用日志、缓存命中、Token、成本、反馈和医生回复。

除 tool call 外，可逐步使用：结构化工作流、多轮症状状态、混合检索、RRF/重排、置信度/拒答、基于证据的回答重组和离线反馈评测。任何生成式步骤都必须位于规则和权限之后，且不能覆盖受控风险结果。

### 4.2 RAG 运行门禁

代码 readiness 通过不等于知识库可用。每个环境必须同时满足：

1. Mongo 目标 collection 数量、内容版本和样本字段正确；当前本地 `medconsult.diseases=8807`。
2. Embedding 模型与导入器一致，当前默认 `BAAI/bge-small-zh-v1.5`、512 维。
3. Milvus 目标 database/collection 存在，可检索源 JSON 的 8807 个唯一主键，维度、metric 和索引参数一致。
4. 执行固定症状探针，验证 Recall@K、引用正确性、高危漏检和超时降级；代码侧已提供 `GET /api/v1/ai/rag/probes` 显式探针接口，关闭时必须在真实 Mongo/Embedding/Milvus 环境取得通过结果。2026-07-19 本地真实依赖验收已通过 3/3；同日用户确认规则种子、离线 Recall@K、引用正确性评测和可追溯 LLM 标准化暂时放弃，后续不作为当前治理 goal 的剩余工作。
5. 缓存键包含知识库内容版本，当前配置项为 `medconsult.ai.rag.knowledge-version` / `RAG_KNOWLEDGE_VERSION`，更新知识后不会命中旧答案。

2026-07-19 本地验收：`medical.data` 已创建并导入，`pymilvus count(*)=8807`，源 JSON 8807 个唯一 `_id` 与 Milvus 可查询 `id` 完全一致。Milvus 2.4.13 REST `get_stats` 在重复 upsert 后可能显示段统计值（当前 `rowCount=8817`），可作为“非空可见”信号，但不能单独证明业务主键数量；以 `count(*)`、主键对账和 `GET /api/v1/ai/rag/probes` 固定症状探针作为关闭证据。2026-07-19 复验 readiness 为 Mongo 8807、Milvus 8807、Embedding 512，固定探针返回 `passed=true`：呼吸道咳嗽、儿童百日咳和胸痛急症均为 UP，其中胸痛样本保留 HIGH 风险和急诊建议。

### 4.3 医疗安全边界

- 急症规则优先，失败时宁可升级人工/急诊，也不能静默给出低风险结论。
- 只读工具默认开放；写操作必须走确定性业务 API、权限、幂等、确认和审计。
- 模型输入最小化，不发送无关身份证号、手机号、完整病历或原始影像。
- AI 输出显示来源、降级状态和不确定性；无证据时追问或拒答。
- 反馈进入坏案例回放、规则/知识修订和离线评测，不只做后台列表。

## 7. UI/UX 基线

视觉方向采用“可访问的临床运营界面”，详见 `design-system/medconsult-assist/MASTER.md`：

- 中性页面背景，医疗蓝主操作、健康绿确认、红/橙风险；风险不只靠颜色表达。
- 患者端强调安全摘要和下一步行动；工作人员端强调扫描、比较和重复操作效率。
- 页面分区不做漂浮卡片；卡片只用于重复记录、模态框和真正的工具面板，圆角不超过 8px。
- 移动端 Drawer 和 44x44px 交互目标作为既有可访问性基线保留；本轮不以移动卡片作为主要关闭口径。
- 工作人员端和管理端宽表页面优先验证桌面缩放稳定性：页面不横向溢出，表格容器可横向滚动，弹窗不越界，错误态可恢复。
- 动效限于 150-260ms 的状态反馈与空间连续性，并尊重 reduced-motion。
