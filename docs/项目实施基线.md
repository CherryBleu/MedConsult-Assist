> 更新日期：2026-07-19
> 用途：结合医疗问诊场景与当前前后端实现，为后续需求、设计、开发、测试和 review 提供统一入口。

## 2. 应用场景

### 2.1 患者：诊前、诊中、诊后连续服务

1. 诊前：注册建档、查看医生与排班、预约缴费、症状问诊和智能分诊。
2. 到院：本人签到、查看候诊状态和叫号通知。
3. 诊中：授权医生查看本人档案、病历和必要影像；AI 只提供可追溯辅助信息。
4. 诊后：查看病历与处方、处方缴费、获取用药提示、申请符合规则的退款。

患者端必须优先展示急症升级建议、风险等级和下一步行动。AI 结果不得伪装成诊断，不得把引用证据或模型话术放在急症提示之前。

### 2.2 医生：接诊与临床决策辅助

1. 查看本人排班和分配给本人的预约，执行接诊状态流转。
2. 在授权范围内查看患者档案、历史病历和影像。
3. 书写结构化病历与处方，提交处方审方。
4. 使用病历摘要、用药分析、影像辅助；医生对最终医疗判断和病历内容负责。

当前医生病历已可书写，但结构化开方、提交审方和处方追踪仍未形成完整前端流程。

### 2.3 药师：审方、调剂和药品安全

1. 按状态处理待审处方，记录通过、驳回和意见。
2. 缴费后执行 FEFO 调剂，保证批次和处方明细可追溯。
3. 处理库存入库、出库、预警和流水。
4. AI 用药分析只提供辅助风险提示，不能覆盖药师的受控规则结果。

当前审方和调剂页面存在，调剂仍是同步跨服务调用；真实支付回调、失败恢复和对账仍需完善。

### 2.4 医院管理员：运营、权限、审计和 AI 治理

1. 管理用户、科室、医生、排班、药品和库存。
2. 管理角色与权限，查看审计日志，追踪关键写操作。
3. 查看 AI 调用、Token、成本、缓存、反馈和医生复审结果。
4. 只可访问职责所需的数据，不因管理员身份默认获得无限患者数据读取权。

RBAC 五表已建但尚未驱动 token scope；审计消费端已建但业务生产端未闭环，不能把空审计表视为“没有风险操作”。

## 3. 当前系统实现

### 3.1 后端服务

| 模块 | 当前职责 | 主要未闭环项 |
| --- | --- | --- |
| gateway | JWT 前置校验、路由、文档聚合 | 无自动化测试；scope 与服务身份仍需集成验证 |
| auth-service | 登录、刷新、注册、用户、服务账号、RBAC DDL | 用户 token scope 已由 `*` 收口为过渡期角色权限点；角色主要依赖 Redis，RBAC 五表尚未驱动鉴权 |
| patient-service | 患者档案与内部查询 | `id_no` 明文，统一 SELF/ASSIGNED SQL 未落地 |
| outpatient-service | 科室、医生、排班、预约、签到、退款 | 叫号/过号/复诊未闭环 |
| medical-record-service | 病历、附件、处方八态流转 | 前端医生结构化开方、患者处方列表/详情/缴费、异步调剂 |
| drug-service | 药品、批次、FEFO、库存流水 | 真实调剂故障恢复和部分写权限需持续验证 |
| notification-service | 通知、审计查询、MQ 消费 | 通用审计生产端/outbox 未接业务服务 |
| ai-service | 分诊、问诊、摘要、用药、报告、影像、反馈、调用日志 | RAG 运行就绪、多轮状态、SSE 前端消费 |
| medconsult-common | Web、安全、MyBatis、Redis、MQ、Feign library | 不可配置生产启动类 |
| data | 医疗知识向量导入工具 | 导入结果必须与运行配置的库/集合/metric 一致 |

### 3.1.1 开发启动基线

`.vscode/launch.json` 只维护可部署 Spring Boot 微服务，不配置 `medconsult-common` 或 `data`。每个启动项必须声明对应 Maven `projectName`、模块目录 `cwd` 和 `classPaths: ["$Runtime"]`，避免 Java 调试器混入 test scope。common 模块允许在 `src/test` 保留测试用 `@SpringBootApplication` 锚点，但不得提供 `main()`；`data` 是导入工具，有 `main()` 也不能作为生产服务挂到 Spring Boot Dashboard。

VS Code / Spring Boot Dashboard 启动项必须使用各自模块目录作为 `cwd`，例如 `ai-service` 对应 `${workspaceFolder}/backend/ai-service`，避免多模块根目录下的相对路径、日志和资源解析歧义。2026-07-19 本地冒烟结果：`ai-service` 可在模块工作目录启动，Tomcat 监听 `8086`，以 `127.0.0.1:8086` 注册到 Nacos，RAG readiness 为 `ready=true`。

### 3.2 前端

前端采用 Vue 3、Pinia、Vue Router、Element Plus 和 Vite，按 PATIENT、DOCTOR、PHARMACY_ADMIN、HOSPITAL_ADMIN 四类角色组织路由。

已完成：路由懒加载、移动 Drawer 主导航与键盘语义、部分响应式表格和错误恢复、AI 会话保留、患者 AI 问诊安全摘要、RAG 证据 disclosure、失败重试、审计/AI 日志移动布局、基础设计令牌、reduced-motion，以及患者 AI 问诊移动端/44px 触控/键盘/axe 聚焦门禁。

仍需完成：医生/药房工作台响应式、影像替代文本、医生开方与患者处方列表/详情/缴费流程、SSE 前端消费、全角色视觉令牌收敛、真实后端 E2E。

## 4. AI 问诊与治理基线

### 4.1 允许的能力

- 本地规则：高危症状、否定词、用药禁忌等确定性规则优先。
- 结构化输出：风险等级、急症建议、可能原因、建议科室和引用证据。
- RAG：Mongo 文档、Embedding、Milvus 向量检索及可追溯 citation。
- 受控工具：服务端白名单只读查询、参数 schema、次数/超时限制、审计 trace。
- 流式交互：SSE 进度、完成和错误事件。
- 多模态：报告文本和医学影像分开处理，医生复审最终结果。
- 治理：调用日志、缓存命中、Token、成本、反馈和医生回复。

除 tool call 外，可逐步使用：结构化工作流、多轮症状状态、混合检索、RRF/重排、置信度/拒答、基于证据的回答重组和离线反馈评测。任何生成式步骤都必须位于规则和权限之后，且不能覆盖受控风险结果。

### 4.2 RAG 运行门禁

代码 readiness 通过不等于知识库可用。每个环境必须同时满足：

1. Mongo 目标 collection 数量、内容版本和样本字段正确；当前本地 `medconsult.diseases=8807`。
2. Embedding 模型与导入器一致，当前默认 `BAAI/bge-small-zh-v1.5`、512 维。
3. Milvus 目标 database/collection 存在，可检索源 JSON 的 8807 个唯一主键，维度、metric 和索引参数一致。
4. 执行固定症状探针，验证 Recall@K、引用正确性、高危漏检和超时降级；代码侧已提供 `GET /api/v1/ai/rag/probes` 显式探针接口，关闭时必须在真实 Mongo/Embedding/Milvus 环境取得通过结果。
5. 缓存键包含知识库内容版本，当前配置项为 `medconsult.ai.rag.knowledge-version` / `RAG_KNOWLEDGE_VERSION`，更新知识后不会命中旧答案。

2026-07-19 本地验收：`medical.data` 已创建并导入，`pymilvus count(*)=8807`，源 JSON 8807 个唯一 `_id` 与 Milvus 可查询 `id` 完全一致。Milvus 2.4.13 REST `get_stats` 在重复 upsert 后可能显示段统计值（当前 `rowCount=8817`），可作为“非空可见”信号，但不能单独证明业务主键数量；以 `count(*)`、主键对账和 `GET /api/v1/ai/rag/probes` 固定症状探针作为关闭证据。

### 4.3 医疗安全边界

- 急症规则优先，失败时宁可升级人工/急诊，也不能静默给出低风险结论。
- 只读工具默认开放；写操作必须走确定性业务 API、权限、幂等、确认和审计。
- 模型输入最小化，不发送无关身份证号、手机号、完整病历或原始影像。
- AI 输出显示来源、降级状态和不确定性；无证据时追问或拒答。
- 反馈进入坏案例回放、规则/知识修订和离线评测，不只做后台列表。

## 5. 测试与质量门禁

### 5.1 后端

- 全量 reactor 测试必须通过，且报告各服务测试数；gateway 不得继续为 0 测试。
- `ai-service` JaCoCo 以全部生产代码为口径，不通过排除 service/controller/client 制造覆盖率。
- `ai-service` 已启用 JaCoCo 门禁：LINE >= 90%，BRANCH >= 80%；`mvn -f backend/pom.xml -pl :ai-service -am verify` 必须执行 report 和 check。
- 2026-07-19 当前验证：`ai-service` 相关 reactor `verify` 通过，Tests run=381，Failures=0，Errors=0，Skipped=0，JaCoCo 输出 `All coverage checks have been met.`；`jacoco.xml` 汇总 LINE 4008/4328（92.61%）、BRANCH 1675/2077（80.65%）。
- 外部客户端必须覆盖超时、空响应、无效 JSON、鉴权失败和降级。
- 权限测试同时覆盖角色、scope、SELF/ASSIGNED owner 和服务身份。

### 5.2 前端

- `npm run build` 与 Playwright 全套通过。
- 视口至少覆盖 375、768、1024、1440；无页面级横向滚动或文字重叠。
- 核心流程必须包含真实后端 E2E，mock E2E 只作为快速回归。
- 键盘、焦点、语义、图像替代文本、reduced-motion 和 axe 形成无障碍门禁。
- 桌面与移动截图用于视觉回归，不能只依据 CSS 静态扫描。

## 6. 数据与隐私基线

### 6.1 测试环境保留集

- 账号：`admin`、`doctor`、`yaofang`、`patient`，密码统一为 `123456`；最多增加 `patient_highrisk`、`patient_refund` 两个确定性场景患者。
- 主数据：固定科室、医生、药品、批次和与账号映射一致的患者档案。
- 知识数据：保留 Mongo diseases、有效 Milvus/etcd/内部 MinIO、Embedding 模型缓存。
- 应用影像 MinIO 与 Milvus 内部 MinIO 是两个数据域，严禁混删。

### 6.2 清理流程

1. 停止 Spring 服务、写入型 E2E、集成测试和 MQ 消费，连续两次计数不再变化。
2. 生成 dry-run 白名单/删除清单，备份 7 个 schema，并导出 Redis/MinIO 清单。
3. 按 AI -> 通知 -> 病历/处方 -> 门诊 -> auth -> patient 的依赖顺序删除非白名单数据。
4. Redis 作废 refresh token，重建白名单 role；清测试幂等键并处理现存 access JWT。
5. 应用 MinIO 只删已删除文件记录对应对象和确认过的孤儿；不删除知识库卷。
6. 重跑确定性 seed，验证 4 个账号登录、owner 权限、无孤儿和 RAG 计数/探针。

禁止在仍有并发写入时清理，也禁止用账号命名启发式批量删除。

## 7. UI/UX 基线

视觉方向采用“可访问的临床运营界面”，详见 `design-system/medconsult-assist/MASTER.md`：

- 中性页面背景，医疗蓝主操作、健康绿确认、红/橙风险；风险不只靠颜色表达。
- 患者端强调安全摘要和下一步行动；工作人员端强调扫描、比较和重复操作效率。
- 页面分区不做漂浮卡片；卡片只用于重复记录、模态框和真正的工具面板，圆角不超过 8px。
- 移动端使用 Drawer，交互目标至少 44x44px，所有功能可用键盘操作。
- 动效限于 150-260ms 的状态反馈与空间连续性，并尊重 reduced-motion。

## 8. 后续改动的固定闭环

1. 逐条核对 `修改建议.md` 第一节并写出状态变化。
2. 明确医疗场景、角色、owner、失败和降级行为。
3. 先写失败测试或可复现基线，再修改实现。
4. 同步接口、数据、架构、前端和测试，不只改单层。
5. 用测试、运行探针、截图或数据对账验证。
6. 更新本文和问题总览，完成一个内聚单元立即 commit，不 push、不创建 PR。
