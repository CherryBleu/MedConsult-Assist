# 智慧医疗系统文档修订建议

## 二、P0：业务闭环与合规硬伤

### 2.1 缺少处方表（处方流转全链路断裂）

**【位置】** `数据库设计文档.md` 2.7 medical_record.prescriptions（TEXT，JSON 字符串）；`接口文档.md` 2.6.1 创建病历把处方塞进 JSON；`需求文档.md` 4.1.4 / 4.2.4

**【问题】**
需求文档 4.2.4 明确要求用药分析基于"当前处方"，但处方当前只是病历里的一坨 JSON，导致：

| 断裂点 | 后果 |
| --- | --- |
| 处方状态无字段 | 药师不知道哪些处方待审，需求里的 PHARMACY_ADMIN 角色无落地点 |
| 审方无落地点 | 处方审核（医疗核心质控环节）无法实现 |
| 处方-药品批次无法精确关联 | `drug_stock_flow.related_record_id` 只能指向病历，无法追溯"某张处方某味药从哪个批次发出" |
| 处方修改无版本 | 医疗文书修改必须留痕，JSON 直接覆盖违规 |
| 退药/退方缺失 | 已发药想退无法处理 |
| 用药分析对象错位 | `ai_medication_analysis.record_id` 关联病历，但分析对象应是处方 |
| 处方明细不可独立查询 | 全塞 JSON，无法做"某药品开了多少张处方"统计 |

**【建议】** 新增两张表，处方从 medical_record 剥离独立成表。

#### 表 1：prescription（处方主表）

| 字段名 | 数据类型 | 说明 |
| --- | --- | --- |
| id | BIGINT | 主键 |
| prescription_no | VARCHAR(32) | 处方编号，如 RX202607060001 |
| record_id | BIGINT | 关联病历 ID |
| patient_id | BIGINT | 患者 ID |
| doctor_id | BIGINT | 开方医生 ID |
| department_id | BIGINT | 开方科室 ID |
| status | VARCHAR(20) | 状态机见下文 |
| pharmacy_pharmacist_id | BIGINT | 审方药师 ID |
| reviewed_at | DATETIME | 审方时间 |
| review_comment | VARCHAR(500) | 审方意见 |
| reject_reason | VARCHAR(500) | 驳回原因 |
| total_fee | DECIMAL(10,2) | 处方总金额 |
| paid_amount | DECIMAL(10,2) | 实付金额 |
| payment_status | VARCHAR(20) | UNPAID/PAID/REFUNDED |
| source | VARCHAR(20) | OUTPATIENT/INPATIENT/EMERGENCY |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |
| deleted | TINYINT | 逻辑删除 |

**处方状态机（8 态）：**
```
DRAFT ──提交──▶ PENDING_REVIEW ──通过──▶ APPROVED ──缴费──▶ PAID
                       │                     │
                     驳回                  退方
                       ▼                     ▼
                   REJECTED             CANCELLED

APPROVED/PAID ──调剂──▶ DISPENSED ──完成──▶ COMPLETED
```

#### 表 2：prescription_item（处方明细表）

| 字段名 | 数据类型 | 说明 |
| --- | --- | --- |
| id | BIGINT | 主键 |
| prescription_id | BIGINT | 处方 ID |
| drug_id | BIGINT | 药品 ID |

**【联动调整】**
- `medical_record.prescriptions`：改为只存**只读快照**用于病历展示，或直接删除该列，处方独立成表后通过 `record_id` 反查。

### 2.3 RBAC 权限模型缺失

**【位置】** `数据库设计文档.md` 2.1 sys_user.role；`需求文档.md` 第 3 节用户角色；全部接口无权限标注

**【问题】** 需求文档 6.1.2 明确要求"使用角色权限控制访问范围"，6.1.4 要求审计，但当前设计存在 4 个结构性缺陷：

1. **角色只定义名字，不定义权限范围**：HOSPITAL_ADMIN 和 PHARMACY_ADMIN 边界模糊（药师管理员能不能管药品？医院管理员能不能管库存？），接口没标权限，谁该调全靠猜。
2. **单 role 字段无法表达多身份**：现实中医生自己生病也是患者、药房管理员兼任 HOSPITAL_ADMIN。当前 sys_user 一个用户只能一个 role，且 patient_id/doctor_id 互斥字段暗示"要么患者要么医生"。
3. **缺少"资源-操作-数据范围"三维权限**：医疗权限不是"按角色"而是按"资源+操作+数据范围"。如医生只能看**自己接诊的**患者病历（行级权限）。需求文档 4.1.1.4 明确"医生只能访问授权就诊范围内的患者档案"，但无落地机制。

### 2.4 微服务边界与内部 Feign 接口未定义

**【位置】** 全部 `接口文档.md`；`需求文档.md` 2.1 技术栈约束（Nacos/Gateway/OpenFeign）

**【问题】** 你本轮提出的核心问题。API 文档全是 `/api/v1/*` 对外接口，**服务之间怎么互相调用完全没设计**。后果：

1. **服务边界形同虚设**：没说怎么拆服务、表归谁独占。开发时会本能"直接连别人的库"，微服务退化成"多模块单体"。
2. **AI_SERVICE 无落脚点**：AI 服务的调用链路（ai-service → patient-service 查档案 → drug-service 查药品）每个 Feign 调用都需内部接口定义，当前一个都没有。
3. **同步 vs 异步无规则**：技术栈有 OpenFeign（同步）和 RabbitMQ（异步），但什么场景用哪个没规则，开发会乱用。
4. **内外接口未隔离**：内部接口和对外接口都挂 `/api/v1`，内部接口会被外部直接访问（安全漏洞），且服务间调用传不过用户 Token。

**【建议】**

#### 建议 1：明确服务拆分与表归属（数据所有权）

**每张表只能归一个服务独占**，别的服务要数据只能 Feign 调：

| 服务 | 独占表 | 对外提供能力 |
| --- | --- | --- |
| auth-service | sys_user, sys_role, sys_permission, sys_role_permission, sys_user_role, sys_service_account, login_log | 用户/角色/权限/鉴权 |
| patient-service | patient | 患者档案 |
| outpatient-service | department, doctor, doctor_schedule, appointment | 科室/医生/排班/预约/签到 |
| medical-record-service | medical_record, attachment, prescription, prescription_item | 病历/处方/附件 |
| drug-service | drug, drug_stock_batch, drug_stock_flow | 药品/库存 |
| notification-service | notification, audit_log | 通知/审计 |
| ai-service | disease_json_knowledge 元数据, ai_chat_session, ai_chat_message, ai_triage_result, ai_medical_summary, ai_medication_analysis, ai_report_text_analysis, ai_image_detection, ai_feedback, ai_call_log | 全部 AI 能力 |
| gateway | （无表） | 路由/限流/鉴权前置 |

#### 建议 2：API 文档新增"内部接口（/internal）"章节

内部接口与对外接口**物理隔离**：
- 对外：`/api/v1/*`，走 Gateway，用户 Token 鉴权
- 对内：`/internal/*`，不走 Gateway（服务间直连），服务 API Key 鉴权

**关键内部接口示例（不完整，需补充）：**

#### 建议 3：定义同步/异步通信选型规则（写入接口约定章节）

> **服务间通信选型**
> - **用 OpenFeign（同步）**：调用方需要立即拿到结果才能继续。如 ai-service 查患者档案做分析、outpatient-service 调 ai-service 拿分诊结果展示给患者。
> - **用 RabbitMQ（异步）**：调用方不需要等待结果，或任务是长耗时/可重试。如审计落库、通知投递、影像检测任务、库存预警计算。
> - **判断标准**：如果"被调方挂了，调用方业务必须失败"，用 Feign；如果"被调方挂了，调用方可以继续（稍后重试）"，用 MQ。

## 三、P0：AI 域关键问题

### 3.1 症状自诊"不调用大模型"与复杂自然语言输入矛盾

**【位置】** `接口文档.md` 3.1 症状自诊问答；`需求文档.md` 4.2.1 "该功能不调用大模型"

**【问题】** 这是文档逻辑最不自洽处：

1. **Embedding 模型本身是大模型**：患者输入"最近三天胸闷、心悸，活动后明显"要变成向量，**必须调用 embedding 模型**（如 bge-m3）。文档需澄清"不调用大模型"实际指"不调用生成式 LLM"，否则技术方案跑不通。
2. **纯向量检索无法处理复杂自然语言**：
   | 难点 | 纯向量检索的问题 |
   | --- | --- |
   | 症状抽取 | "活动后加重"是修饰语，向量检索难精确命中 symptom 字段 |
   | 否定语义 | "我没有发烧"——向量检索会把"发烧"也召回 |
   | 时间理解 | "三天"是持续时间，影响 risk_level，向量检索忽略 |
   | 高危识别 | "胸痛+冷汗"是急危重症，靠向量 score 排不出来，需规则 |
   | 同义词 | "心慌" vs "心悸"，依赖 embedding 质量 |
3. **返回示例 answer 文本过于通顺**：纯模板拼接很难生成示例中那样连贯的句子。

**【建议】** 已确认方案：**规则 + Embedding 为核心，允许叠加生成式 LLM**（输入标准化 / 回答润色），但回答必须可追溯到疾病 JSON 命中片段。

#### 对外暴露与使用边界

- 对外接口固定为 `POST /api/v1/ai/symptom-chat`，由 Gateway 路由到 ai-service 的 `POST /ai/symptom-chat`；不得将 `/internal/*` 路径暴露给客户端。
- 患者完成登录并携带有效 Bearer Token 后即可调用，不以预约、挂号、接诊或病历记录作为前置条件。
- 接口仅授权 `PATIENT` 角色，权限码 `ai:symptom-chat`，数据范围 `SELF`。患者 ID 以 JWT 登录身份绑定的 `patient_id` 为准，请求体中的 `patientId` 应移除或仅作一致性校验，禁止代他人发起自诊。
- 自诊会话和消息按当前登录患者归属并记录审计上下文；结果仅供健康咨询参考，不能替代医生诊断。

#### 文档措辞修订

> **2026-07-16 修订**：原"不调用大模型/不调用生成式 LLM"为硬约束，现放宽为——**症状自诊问答以本地 Embedding 模型（如 bge-m3）向量召回 + 规则引擎为核心；当规则难以覆盖口语化输入或需自然语言润色时，允许叠加生成式 LLM 做输入标准化与回答重组，但回答文本必须能追溯到疾病 JSON 命中片段，不得引入模型自由生成的医学论断**。三份文档（需求/接口/架构）已同步此措辞。

#### 完整处理链路

```
患者输入
  ├─① 本地 Embedding 向量化（bge-m3 等，可离线或在线）
  ├─② 向量召回 topK（Milvus/Qdrant，按 disease_json 字段切片）
  ├─③ 规则过滤：否定词剔除、高危症状强制提权
  ├─④ 规则判定 risk_level（CRITICAL 规则优先）
  └─⑤ 模板拼装 answer + citations + suggestedDepartments
```

### 3.2 ai_imaging_detection 文本报告与影像文件混表

**【位置】** 原 `数据库设计文档.md` 3.8 ai_imaging_detection；`接口文档.md` 3.5

**【问题】** 两种输入代表完全不同的检测能力，混在一张表一个接口：

| 字段 | 实际含义 | 需要的模型 | 难度 |
| --- | --- | --- | --- |
| report_text | 医生/设备**已写好的文字报告** | NLP 文本分类 | 低 |
| image_urls | 原始 DICOM/CT **影像文件** | 医学视觉模型 | 极高 |

1. **业务意义冲突**：report_text 已是医生结论，再让 AI"检测异常"意义何在？要么检测报告漏写的病灶（NLP），要么对原始影像重新分析（视觉），是完全不同的产品。
2. **外部模型调用不一致**：接口 externalModel.model="chest-ct-abnormal-detector" 看名字是图像模型，但请求又带 reportText，到底传哪个给外部模型？
3. **数据类型不统一**：image_urls 是 JSON 数组，report_text 是 TEXT，一次任务通常只有一种，另一字段长期 null。
4. **文件存储链路断裂**：DICOM 单文件几十 MB，attachment 表 file_url 只是 URL，**没有文件上传接口**（POST /attachments 只建元数据记录）。

## 四、P1：技术栈核心能力落地

### 4.1 SSE 流式接口缺失（JDK21 虚拟线程 + Spring AI）

**【位置】** `需求文档.md` 2.1 "使用 JDK 21 虚拟线程承载 AI 大模型长连接、SSE 流式输出"；`接口文档.md` 全部 AI 接口同步返回

**【问题】** 技术栈明确强调 SSE 流式输出，但所有 AI 接口都是同步阻塞。Spring AI 的核心卖点之一是流式输出，没用上。

**【建议】** 为生成式 AI 接口补充 SSE 版本：

```yaml
GET /ai/medical-record-summary/stream?recordId=xxx
Accept: text/event-stream
→ 逐 token 推送，最后推送 [DONE]

GET /ai/medication-analysis/stream
GET /ai/triage/stream
```

适用：病历摘要、用药分析、智能分诊。**症状自诊不流式**（非生成式）。

---

### 4.2 RabbitMQ 异步场景未在接口层体现

**【位置】** `需求文档.md` 2.1 "RabbitMQ 用于 AI 摘要、影像检测、通知投递、AI 调用日志等异步场景"

**【问题】** 所有接口都同步返回，MQ 存在感为零。

**【建议】** 明确异步语义，定义关键队列：

| Exchange / Queue | 用途 |
| --- | --- |
| ai.imaging.exchange → ai.image.detect.queue | 影像检测异步任务 |
| notification.exchange → notification.send.queue | 通知异步投递 |
| log.exchange → audit.log.queue | 审计日志异步落库 |
| log.exchange → ai.calllog.queue | AI 调用日志异步落库 |
| drug.event.exchange → drug.stock.alert.queue | 库存预警异步计算 |

接口文档需补充"异步任务"语义说明（如影像检测提交即返回 PENDING）。

---

### 4.3 Redis 分布式锁与限流未体现

**【位置】** `需求文档.md` 2.1 "Redis 用于...分布式锁"

**【问题】** 高并发号源抢占、库存扣减场景未体现分布式锁；AI 接口无限流（烧钱风险）。

**【建议】** 在接口约定章节补充：

| 场景 | 实现 |
| --- | --- |
| 预约抢号 | Redis Lua 原子扣减 `schedule:{id}:quota` |
| 药品出库 | 分布式锁 `lock:drug:{id}:stock` |
| 支付回调 | 幂等锁 `lock:payment:{paymentNo}` |
| AI 接口限流 | 滑动窗口 `ratelimit:ai:{userId}`（防止成本失控） |
| 号源缓存 | `schedule:available:{date}` 缓存可预约号源 |

### 4.4 Function Calling 函数未定义

**【位置】** `接口文档.md` 3.4 用药分析；`数据库设计文档.md` 3.7 ai_medication_analysis.function_trace

**【问题】** 用药分析提到 Function Calling，function_trace 记录了 queryDrugInteractions，但**没有定义 Function 的 schema**。且 `drug.contraindications`、`interactions` 是 TEXT 自由文本，**结构化程度不足以支撑 Function 调用**。

**【实施状态】** 2026-07-18 已落地“受控 tool_call/functionTrace”与服务端 schema 白名单：`MedicationFunctionService` 由服务端固定执行 `queryPatientAllergies`、`queryCurrentMedications`、`queryDrugRiskInfo`、`queryDrugContraindications`、`queryDrugInteractions`，trace 统一包含 `type=tool_call`、`schemaVersion=medication-tool-trace-v1`、`toolName/functionName`、`inputSummary`、`status`、`resultSummary`、`errorCode`；未知工具名、未知状态、患者/药品入参摘要不匹配会被拒绝；药品风险 Feign 按 drugId 去重，失败以 FAILED trace 显式降级；`MedicationAnalysisService` 已强制以服务端 `FunctionResult` 为受控基线，LLM 返回的 LOW/空风险/伪造 trace 不能覆盖服务端风险与 trace。仍未让 LLM 自主发起 native tool_call（医疗安全上需先保证服务端白名单与审计）。

**【建议】**
1. 补充 Function 定义章节：

| Function | 参数 | 数据源 |
| --- | --- | --- |
| queryPatientAllergies | patientId | patient.allergies |
| queryCurrentMedications | patientId | prescription_item（status≠CANCELLED） |
| queryDrugContraindications | drugId | drug.contraindications（需结构化） |
| queryDrugInteractions | drugAId, drugBId | drug.interactions（需结构化） |

2. drug 表结构调整：

```diff
  drug
- contraindications TEXT
- interactions TEXT
+ contraindications JSON   -- 结构化：[{condition:"严重低血压", level:"ABSOLUTE"}, ...]
+ interactions JSON        -- 结构化：[{drug_code:"XXX", effect:"增加出血风险", level:"MEDIUM"}]
```

## 五、P1：字段一致性与安全

### 5.1 字段一致性问题汇总

**【deleted 字段不统一】** 通用约定声明 deleted 为通用字段，但只有 patient、attachment 两表实际有。需补齐所有业务实体表：sys_user、department、doctor、doctor_schedule、appointment、medical_record、drug、drug_stock_batch、notification、prescription、prescription_item、所有 ai_* 结果表。**流水表**（drug_stock_flow、ai_chat_message、ai_call_log、audit_log、login_log）不加 deleted（只追加）。

**【枚举不对齐】**

| 枚举 | 现状 | 统一为 |
| --- | --- | --- |
| userRole | DB 无 AI_SERVICE，接口有 | DB 说明同步补 AI_SERVICE（但 AI_SERVICE 走 sys_service_account，不在 sys_role） |
| paymentStatus | 已对齐：UNPAID/PAID/REFUNDING/REFUNDED；退款状态仅由退款流程写入 | 维持接口约束，禁止 payment patch 伪造退款状态 |
| aiCallType | DB 用 SUMMARY；接口枚举用 MEDICAL_RECORD_SUMMARY；查询参数又用 type | 统一 SYMPTOM_CHAT/TRIAGE/MEDICAL_RECORD_SUMMARY/MEDICATION_ANALYSIS/IMAGING_DETECTION/REPORT_ANALYSIS |
| 新增 prescriptionStatus | 无 | DRAFT/PENDING_REVIEW/APPROVED/REJECTED/PAID/DISPENSED/COMPLETED/CANCELLED |

**【接口字段无对应列】**

| 接口字段 | 缺失列 | 处理 |
| --- | --- | --- |
| appointment.source | 无 | 补 source 列 |
| 取消预约 operatorType | 无 | 补 cancel_operator_type、cancel_operator_id |
| 更新支付 paymentNo/paidAmount | 无 | 补 payment_no、paid_amount |
| 药品 stockQuantity | 无 | drug 补 current_stock |

**【created_at/updated_at 不统一】** notification（标记已读会更新）仍需复核；影像检测已拆为 `ai_image_detection`，且状态变更表已包含 `updated_at`。

**【JSON 字段建议改 JSON 类型】** MySQL 8.0 支持 JSON 类型，doctor.specialties、medical_record.prescriptions/initial_diagnosis、drug.contraindications/interactions 建议从 VARCHAR/TEXT 改为 JSON，便于校验和查询。

## 六、P2：业务流程与规范优化

### 6.1 就诊主流程不闭环

完整链路 `预约→支付→签到→候诊→就诊→开方→审方→缴费→取药→复诊` 当前缺：
- 签到接口已补 `POST /appointments/{id}/check-in`（患者本人、已支付、`BOOKED -> CHECKED_IN`）
- 叫号/过号机制（queue_no 有，过号处理无）
- 复诊预约关联（首次 vs 复诊）
- 状态流转规则（哪些状态可流转到哪些）

**【建议】** 继续补叫号/过号机制；在 appointment 表补 `visit_type`（FIRST/FOLLOW_UP）和 `parent_appointment_id`（复诊关联）。

### 6.2 退费流程缺失

**【实施状态】** 2026-07-17 已补齐挂号费退款最小闭环：新增 `refund_order` 表、`POST /appointments/{id}/refund`、RefundRequest/RefundResponse、RefundService/Controller、Redis 锁 + `appointment_id` 唯一约束幂等、患者端“申请退款”入口和 mock E2E；同时 `PATCH /appointments/{id}/payment` 已禁止直接写入 REFUNDING/REFUNDED。

原问题：paymentStatus 有 REFUNDING/REFUNDED，但无退款接口、无退款单表、退款规则未定义。

**【后续建议】** 当前实现为挂号费全额 MOCK 退款；真实支付渠道接入时需补支付回调验签、失败重试、人工退款与对账任务。部分退/全退规则仍需结合正式支付产品方案明确。

### 6.3 FEFO 出库策略未定义

医疗强制 **FEFO（近效期先出）**，但 `POST /drugs/{drugId}/stock/outbound` 只传 quantity，未指定批次选择策略。

**【建议】** 接口增加可选 `batchStrategy=FEFO|FIFO|MANUAL` 参数；默认 FEFO；drug-service 内部按 expire_date 升序选批，多批次混合出库时一条出库流水对应多个批次明细。

### 6.4 检查检验申请缺失

病历写"完善心电图检查"，attachment.biz_type 有 EXAM_REPORT，但**无检查检验申请单表**，申请→预约→执行→报告回传全链路缺失。影像检测 AI 的影像从哪来无源头。

**【建议】** 后续迭代新增 `exam_request`（检查申请单）表，打通检查检验闭环。当前 P2 可暂缓。

### 6.5 GET 请求不应携带请求体

接口文档所有 GET 接口都写了 `**请求示例** {}`。HTTP GET 不应有 body。建议移除空请求体示例，或改为"无请求体，参数走 query string"。

### 6.6 分页默认值不统一

接口约定 pageSize 默认 10，但科室列表示例 pageSize=20。建议统一默认 10，最大 100。

### 6.7 病历/影像附件与 AI 任务关联

附件表 attachment 当前可关联病历，但 AI 影像检测任务与附件无直接关联。建议 ai_image_detection.attachment_ids（JSON）或 ai_image_detection 通过 record_id 间接关联。
