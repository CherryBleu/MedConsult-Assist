3. 向量化

	为了让 Embedding 模型最大程度地捕捉“症状-疾病”的关联，我们需要将 JSON 拆分为两部分：

	- **用于生成向量的文本 (Text Content)：** 只提取与疾病定义和症状强相关的字段。包括 `name`（疾病名称）、`symptom`（症状列表）和 `desc`（疾病描述）。
	- **用于过滤和附加信息的元数据 (Metadata)：** 将科室、治疗方法、药品、易感人群等结构化字段存为 Metadata。当检索到该疾病时，这些 Metadata 会随着文本一起提供给 LLM，用于生成最终的全面回答。

	注意两种json格式

4. ###  LLM 优化层的核心工作流程

	当用户输入一个模糊或复杂的医疗咨询时，LLM 需要同时完成两件事：

	1. **生成用于向量检索的文本**：提取或生成针对 `name`、`desc`、`symptom` 的纯医学语义描述。
	2. **提取用于 Qdrant 过滤的元数据**：匹配 `cure_department`、`yibao_status` 等字段。

	**不限制 LLM 必须输出标准的字典值，而是让它输出“原始意图”，然后在代码层用传统的搜索算法去匹配标准值。**

	- **工作流：**

		1. **LLM 自由提取**：去掉 Prompt 里的 `Literal` 枚举限制，让 LLM 提取用户原话中的含义。例如用户问“小孩子咳嗽挂什么科”，LLM 提取出 `cure_department: "小儿科"`（这在你的数据库里是不存在的，你库里叫“儿科”）。

		2. **本地词典映射**：拿到 `"小儿科"` 后，不直接去 Qdrant 搜，而是用轻量级的模糊匹配算法（如快速的 BM25、FuzzyWuzzy 字符串相似度，或者本地的一个小 Embedding 模型），拿着 `"小儿科"` 去你的“全量科室标准字典”里匹配，得出最高相似度的是 `"儿科"`。

		3. **Qdrant 过滤**：拿着标准映射后的 `"儿科"` 放入 Qdrant 的 Filter 中。

			向量检索依靠的是语义相似度（Cos/Dot/L2），所以 LLM 生成的文本不需要和数据库里的字眼一模一样，但**必须知道数据库里有什么类型的信息**。
