# Code Review（迭代式 · 发现即修复）

本仓库是 Spring Cloud Alibaba 多模块微服务项目（MedConsult-Assist）。本 skill 对**本地代码**做迭代式 review：结合 `git` 的 commit / diff 锁定范围，每一轮从**安全性 / 性能 / 逻辑 / 潜在问题**四个维度排查，**发现问题就当场修复**，然后对修复结果做下一轮 review，直到这一批代码干净或达到 5 轮上限。

## 核心行为约束（必读）

1. **结合 git**：每一轮开始先用 git 锁定要 review 的文件，不要凭记忆。review 的对象是 git 能看到的改动。
2. **迭代 loop 3–5 轮**：第 1 轮起步，最多 5 轮。每一轮 = 一次完整 review +（发现则）修复 + 一份总结。
3. **发现即修复**：找到问题当场改（不是只列清单）。下一轮 review 的是**修复后的代码**。这是收敛的关键。
4. **可以提前停止**：某一轮确实没发现任何问题（四个维度都干净），即可停止，不必硬凑到 5 轮。停止时说明"第 N 轮无新发现，已收敛"。
5. **每轮都要总结**：无论改没改，都要输出本轮总结（见下方格式）。这是硬性要求。
6. **到 5 轮仍未收敛**：停止修复，把**遗留问题清单**汇报给用户，由用户决定后续，不要无限循环。

## 第 0 步：确定范围（每轮固定做一次）

默认范围 = **当前分支相对 `main` 的全部改动**。先用命令确认：

```bash
# 当前分支 vs main 的改动文件清单
git diff --name-only main...HEAD
# 完整 diff（代码级，review 的主要输入）
git diff main...HEAD
# 本次分支涉及的关键 commit
git log --oneline main..HEAD
```

按用户的覆盖参数调整范围：

| 用户说的 | 范围命令 |
| --- | --- |
| "review 最近的提交" / "review 这次" / "review HEAD" | `git show HEAD`（单次 commit） |
| "review 最近 N 次" | `git log -p -N` |
| "review <commit-sha>" | `git show <sha>` |
| "review 当前分支" / 不指定（默认） | `git diff main...HEAD` |
| "review 未提交的改动" / "review 工作区" | `git diff` + `git diff --staged` |
| 指定文件/目录路径 | 只 review 这些路径 |

> 确定范围后，**只读改动过的源文件**（Controller / Service / Mapper / DTO / 配置等），不要扩散去通读全仓库。范围外的可疑点可一句话提示，不展开。

**Spring Cloud 专项**：本仓库的对外接口都有 Postman 集合（`postman/` 目录，提交里常出现"共 N 接口"）。凡 review 到对外 Controller / 内部 Feign 接口时，要顺带确认：
- 接口路径、HTTP 方法、入参 DTO、出参结构是否与 `docs/接口文档.md` 一致；
- 是否已在 Postman 集合里登记（新增/改动的接口要让集合保持可测）；
- **确保接口"完全可用"**：入参校验（Bean Validation）、鉴权、异常处理、事务边界、幂等性（涉及写操作）、跨服务 Feign 调用的 fallback 都齐备，不是只编译通过。

## 第 1 步起：迭代 review（loop 3–5 轮）

每一轮按下面四个维度过一遍**当前范围内的代码**。每发现一条问题，**当场修复**（Edit 改文件），修完继续，直到这一轮扫完。轮与轮之间不要换范围——固定 review 这一批改动，只是不断深入。

### 四个维度（每轮都过）

#### ① 安全性
重点找真实可利用的高危项，别拿理论问题凑数：

- **密码 / 凭证**：是否明文存储？是否用了项目约定的哈希（如 BCrypt）？是否有硬编码的密钥、token、数据库密码、Nacos 凭证落在代码或配置里？
- **SQL 注入**：MyBatis-Plus 用 `${}` 拼接而非 `#{}`？动态 `ORDER BY` / `IN` 拼接用户输入？`QueryWrapper` 里塞了未校验的用户串？
- **越权**：接口有没有从登录态/`userNo` 取身份，而不是信任请求体里的 `userId`？内部接口（Feign）有没有被外部网关误暴露？
- **敏感信息泄露**：异常把堆栈/SQL 原文吐给前端？日志打了密码、身份证、手机号明文？返回 DTO 带了不该带的字段（如密码哈希）？患者信息是否脱敏？
- **校验缺失**：入参 DTO 有没有 Bean Validation（`@NotBlank`/`@Valid` 等）？路径参数有无范围校验？
- **并发安全**：库存、抢号这类读改写有没有 Redis 锁 / DB 乐观锁 / `FOR UPDATE`？锁有没有正确释放（`try-finally`）？

#### ② 性能
找**这一批改动里**真正值得改的，不搞过度优化：

- **N+1 查询**：循环里查 DB / 调 Feign？能不能 `selectBatchIds` / 批量 Feign / 一次 join？
- **复杂度**：有无更深的嵌套循环、对大集合做多次全量遍历、在循环里重复计算可提到循环外的量？时空复杂度能否再降一档？
- **查询效率**：`SELECT *`？没用上索引的查询条件？分页里没加 `LIMIT`、深分页？缓存（Redis）该加的热点没加、或加了但没设过期/防穿透？
- **批量写**：循环单条 `INSERT`/`UPDATE` 没用 `saveBatch`/`updateBatchById`？
- **资源**：IO / 远程调用没设超时？连接/流没关（`try-with-resources`）？

#### ③ 逻辑正确性
- **状态机**：处方/订单这类多态流转，状态校验有没有漏（非法跳转、重复操作）？状态字段读写与文档/枚举一致？
- **边界与空值**：判空对了吗（`equals` 常量在前）？空集合 / null 返回 / 除零 / 数组越界 / 分页页码 0 或负？
- **事务**：`@Transactional` 标了但（a）自调用没生效、(b) 默认只回滚 `RuntimeException`、(c) Feign 调用跨了事务边界、(d) 异常被 catch 吞掉导致不回滚？
- **幂等**：写接口（尤其 MQ 消费者）有没有 `idemKey` / 去重表 / Redis 标记？重试会不会产生脏数据？
- **并发与时序**：异步/MQ 之间、Feign 调用之间的执行顺序假设是否成立？`@Async` / `@EventListener` 时机对不对？
- **返回值与契约**：方法返回与调用方期望一致？接口出参与 `接口文档.md` 一致？错误码语义对？

#### ④ 潜在问题
- **健壮性**：外部依赖（Redis/MQ/Nacos/下游服务）不可用时降级路径在不在？`fallback` 写得对不对？
- **异常处理**：`catch (Exception e)` 吞异常？只 log 不抛？自定义业务异常体系用对了吗？
- **资源泄漏与配置**：线程池/连接池参数、Redis key 过期、定时任务重复触发、@Scheduled 多实例重复执行？
- **配置**：`application.yml` 里写的值（超时、重试、连接数）合不合理？环境相关配置有没有写死？
- **可维护性**（轻量提示即可）：魔法值、重复代码块、命名误导——只点出，不强制改。

> 四个维度是清单不是顺序，一轮内可以并行思考。对每条发现，先判断它**是否落在本次 diff 范围内**——范围外的老代码问题，一句话提示，不展开修。

## 第 N 步：每轮总结（硬性要求）

无论这一轮改没改、停没停，都要给一份**本轮总结**，用这个格式：

```
## 第 N 轮总结
- 扫描范围：<文件清单或一句话描述>
- 本轮发现：<条目，按维度归类；无则写"无新发现">
  - [安全性] xxx —— 已修复：<改了什么，一句话>
  - [性能]   xxx —— 已修复：...
  - [逻辑]   xxx —— 已修复：...
  - [潜在]   xxx —— 已修复：...
- 验证：<编译/测试/接口可用性确认情况，没跑就如实说"未运行">
- 下一步：<进入下一轮 / 已收敛停止 / 达到上限，汇报用户>
```

修复后要**验证**：能编译就 `./mvnw -q -pl <模块> -am compile`（或项目惯用命令），有相关测试就跑测试，涉及接口的确认其可调用性。验证失败要如实写在总结里，不能宣称"已修复"。

## 停止条件（明确）

- **提前停止（已收敛）**：某一轮四个维度都没新发现 → 输出该轮总结（"第 N 轮无新发现，已收敛"）后结束。
- **到 5 轮上限**：第 5 轮结束仍没收敛 → 停止修复，**汇报用户**：列出仍未解决的问题清单（带维度、文件、风险等级、建议），交给用户决定，不要进第 6 轮。
- 不要在一轮里假装把所有问题都"一次改完然后下一轮就空"——前几轮要真正深入查（性能/逻辑/潜在问题往往要深读调用链才暴露），靠迭代层层深入，不是一轮扫完。

## 整体收尾

所有轮次结束后（无论收敛还是到上限），给一份**总览**：

```
## Review 总览
- 范围：<分支/commit/文件>
- 总轮数：<N / 最多 5>
- 结果：<已收敛 / 达上限未完全收敛>
- 累计修复：<X 条，按维度统计>
- 接口可用性：<确认了哪些对外/内部接口可调用；哪些存疑>
- 遗留问题（若有）：<清单 + 建议，等用户决策>
```

## 常用命令速查

```bash
git diff --name-only main...HEAD        # 默认范围文件清单
git diff main...HEAD                    # 默认范围完整 diff
git show HEAD                           # 单次 commit
git log --oneline main..HEAD            # 分支上的 commit 列表
git diff --stat main...HEAD             # 改动行数概览，快速定位大改动文件
./mvnw -q -pl <module> -am compile      # 编译某模块（修完验证用）
```

## 写作约定

- 总结用中文（与本仓库 commit / docs 一致）。维度标签可用 `[安全性]`/`[性能]`/`[逻辑]`/`[潜在]`。
- 每条发现要给 `文件:行号`，方便点击跳转。
- 风险评级（高/中/低）用于"到上限汇报用户"那份清单；正常迭代里可省。
- 优先修高/中危，低危与纯风格问题只提示不强行改。
